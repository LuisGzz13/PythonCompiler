Análisis semántico · Cubo + Directorio de Funciones + Tablas de Variables · Luis Manuel Gonzalez Martinez · A01722501

Visión general

Etapa 2 agrega una pasada más sobre el árbol de parseo que produjo Etapa 1: el analizador semántico. Su trabajo es recorrer el árbol, registrar cada función en un Directorio de Funciones y cada variable en su Tabla de Variables, y reportar violaciones (en esta etapa, solo duplicados). Además se diseña el Cubo Semántico que en Etapa 3 servirá para validar tipos en expresiones.

Pipeline actualizado:
archivo.patito → PatitoLexer → PatitoParser → SemanticAnalyzer (listener) → reporta OK / FAIL

Lo nuevo en esta etapa son 7 archivos C++ dentro de src/semantic/, más modificaciones a src/main.cpp, CMakeLists.txt y tests/run_tests.sh para integrar todo. Este walkthrough explica los archivos nuevos línea por línea, y muestra solo las partes añadidas de los archivos modificados.

El módulo está organizado en cuatro unidades:

  • SemanticTypes.h — los enum class de tipos y operadores que el resto del módulo usa.
  • SemanticCube.{h,cpp} — el cubo semántico, una función que dada (tipo, operador, tipo) devuelve el tipo resultado.
  • SymbolTable.{h,cpp} — las estructuras que almacenan funciones y variables.
  • SemanticAnalyzer.{h,cpp} — el listener que recorre el árbol y llena las estructuras anteriores en los puntos neurálgicos.

#pragma once evita que el header se incluya dos veces en la misma unidad de compilación (alternativa moderna a los include guards #ifndef ... #define ... #endif). Es soportado por todos los compiladores que usamos (clang, gcc, MSVC).

<string> se necesita para las firmas tipoToStr y opToStr que devuelven std::string.

8namespace patito {

Todo el código semántico vive en el namespace patito. Esto evita colisiones con símbolos del runtime ANTLR (que vive en antlr4) o de la stdlib. En main.cpp usaremos patito::SemanticAnalyzer explícitamente.

10// Tipos que maneja el analizador. 11// - ENTERO / FLOTANTE: tipos del lenguaje Patito. 12// - NULA: solo para tipo de retorno de funciones. 13// - BOOL: tipo interno, resultado de operadores relacionales. 14// - ERROR: combinacion invalida en el cubo semantico. 15enum class Tipo { 16 ENTERO, 17 FLOTANTE, 18 NULA, 19 BOOL, 20 ERROR 21};

Declara los 5 tipos que el analizador maneja. Uso enum class (no el viejo enum de C) para que los valores estén scopados: hay que escribir Tipo::ENTERO, no solo ENTERO. Esto evita colisiones con macros y obliga a tipar explícitamente.

  • ENTERO, FLOTANTE — los dos tipos del lenguaje, los que el usuario declara.
  • NULA — solo válido como tipo de retorno de función. Las variables no pueden tener este tipo.
  • BOOL — tipo interno. El usuario no lo escribe; el cubo lo devuelve cuando dos operandos numéricos se comparan con >, <, == o !=. Sirve para Etapa 3 cuando una expresión así se use en si (...) o mientras (...).
  • ERROR — sentinela. El cubo lo devuelve para combinaciones inválidas (p. ej. sumar dos NULA).
23// Operadores del lenguaje cubiertos por el cubo semantico. 24enum class Operador { 25 MAS, // + 26 MENOS, // - 27 POR, // * 28 ENTRE, // / 29 MAYOR, // > 30 MENOR, // < 31 DISTINTO, // != 32 IGUAL2, // == 33 ASIGNA // = 34};

Los 9 operadores que el cubo reconoce. Los nombres coinciden con los tokens del lexer (MAS, MENOS, etc., en Patito.g4) salvo ASIGNA, que en el lexer se llama IGUAL pero aquí le di un nombre más descriptivo porque "IGUAL" en el contexto semántico se confunde con IGUAL2 (==).

Cada operador se usará como uno de los tres ejes del cubo: (tipoIzq, Operador, tipoDer) → Tipo.

Funciones libres (no miembros) que convierten un valor del enum a su nombre humano. Esto es lo que el --dir y los mensajes de error usan para imprimir "entero" en vez de "0". Solo declaradas aquí; el cuerpo está en SemanticCube.cpp.

Interfaz pública del cubo semántico: una clase con un solo método estático que responde "(izq, op, der) → resultado".
1// Cubo Semantico de Patito. 2// Dada una pareja de tipos y un operador, devuelve el tipo del resultado 3// (o Tipo::ERROR si la combinacion no es valida). 4// 5// Reglas: 6// - Aritmetico (+ - * /): entero op entero -> entero 7// cualquiera con flotante -> flotante 8// - Relacional (> < != ==): cualquier combinacion numerica -> BOOL 9// - Asignacion (=): entero = entero -> entero 10// flotante = entero -> flotante (ensanchamiento implicito) 11// flotante = flotante -> flotante 12// entero = flotante -> ERROR (perdida de precision) 13// 14// Etapa 2 solo construye el cubo; Etapa 3 lo aplicara al generar cuadruplos.

Comentario de cabecera con la tabla de reglas del cubo. Ponerlo aquí (en lugar de en el .cpp) es deliberado: el usuario que #include "SemanticCube.h" ve inmediatamente qué resultados esperar sin tener que abrir la implementación.

16#pragma once 17 18#include "SemanticTypes.h"

tipoToStr y opToStr

5std::string tipoToStr(Tipo t) { 6 switch (t) { 7 case Tipo::ENTERO: return "entero"; 8 case Tipo::FLOTANTE: return "flotante"; 9 case Tipo::NULA: return "nula"; 10 case Tipo::BOOL: return "bool"; 11 case Tipo::ERROR: return "error"; 12 } 13 return "?"; 14}

Convierte un valor del enum a su representación en texto. El switch cubre los 5 valores; el return "?" al final es defensivo: si en el futuro se agrega un nuevo valor al enum y se olvida agregar el case, esta función al menos no compila silenciosamente con un valor basura (devuelve "?"), avisándo el problema.

16std::string opToStr(Operador op) { 17 switch (op) { 18 case Operador::MAS: return "+"; 19 case Operador::MENOS: return "-"; 20 case Operador::POR: return "*"; 21 case Operador::ENTRE: return "/"; 22 case Operador::MAYOR: return ">"; 23 case Operador::MENOR: return "<"; 24 case Operador::DISTINTO: return "!="; 25 case Operador::IGUAL2: return "=="; 26 case Operador::ASIGNA: return "="; 27 } 28 return "?"; 29}

Análogo a tipoToStr, pero para operadores. Devuelve el lexema original ("+", "-", etc.). Útil para mensajes de error en Etapa 3 cuando se reporten incompatibilidades de tipos en una expresión específica.

resultado()

31// Solo aceptamos combinaciones de tipos numericos (ENTERO/FLOTANTE) 32// en los operadores. NULA y BOOL no son operandos validos. 33static bool esNumerico(Tipo t) { 34 return t == Tipo::ENTERO || t == Tipo::FLOTANTE; 35}

Helper privado al archivo (static a nivel de archivo significa "internal linkage": no se exporta a otros .cpp). Centraliza el chequeo de "este tipo es válido como operando".

El razonamiento: ni NULA (tipo de retorno, no de operando) ni BOOL (resultado intermedio) deberían aparecer a los lados de un operador. Si lo hacen, es un error.

37Tipo SemanticCube::resultado(Tipo izq, Operador op, Tipo der) { 38 if (!esNumerico(izq) || !esNumerico(der)) { 39 return Tipo::ERROR; 40 }

Apertura de la función pública. La primera línea filtra cualquier combinación que involucre operandos no-numéricos. Esto reduce los casos a manejar después: cuando llegamos a la línea 42 sabemos que tanto izq como der son ENTERO o FLOTANTE.

42 const bool algunoFlot = (izq == Tipo::FLOTANTE) || (der == Tipo::FLOTANTE);

Atajo útil: "¿alguno de los dos operandos es flotante?". Lo uso para decidir la promoción en operadores aritméticos (un flotante en cualquier lado vuelve el resultado flotante).

44 switch (op) { 45 // Aritmeticos: el flotante "gana" si aparece en cualquiera de los dos. 46 case Operador::MAS: 47 case Operador::MENOS: 48 case Operador::POR: 49 case Operador::ENTRE: 50 return algunoFlot ? Tipo::FLOTANTE : Tipo::ENTERO;

Los cuatro operadores aritméticos comparten regla y por eso comparten case (técnica clásica del "switch caída"). Si alguno de los operandos es flotante, el resultado lo es. Si los dos son enteros, el resultado es entero.

Esto refleja la matemática esperada: 3 + 2.5 = 5.5 (flotante), 3 + 2 = 5 (entero).

52 // Relacionales: cualquier combinacion numerica produce BOOL. 53 case Operador::MAYOR: 54 case Operador::MENOR: 55 case Operador::DISTINTO: 56 case Operador::IGUAL2: 57 return Tipo::BOOL;

Los cuatro relacionales (>, <, !=, ==) devuelven BOOL sin importar la combinación de operandos numéricos. 3 < 2.5 es perfectamente válido y produce BOOL (false, en realidad), igual que 1 == 1.

El BOOL resultante es lo que en Etapa 3 alimentará al si y al mientras — esos estatutos exigirán que su expresión sea BOOL.

59 // Asignacion: izq es el destino, der es la fuente. 60 // entero = entero -> OK 61 // flotante = entero -> OK (ensanchamiento) 62 // flotante = flotante -> OK 63 // entero = flotante -> ERROR (perdida) 64 case Operador::ASIGNA: 65 if (izq == Tipo::ENTERO && der == Tipo::FLOTANTE) { 66 return Tipo::ERROR; 67 } 68 return izq; 69 } 70 return Tipo::ERROR; 71}

La asignación es asimétrica (los lados no son intercambiables, a diferencia de +). El único caso de error es asignar un flotante a un entero, porque eso perdería la parte decimal. Las otras tres combinaciones son aceptadas:

  • entero = entero: trivialmente OK, resultado entero.
  • flotante = entero: el entero se promueve a flotante (ensanchamiento implícito, sin pérdida).
  • flotante = flotante: OK.

El return izq devuelve el tipo del destino, que es lo natural para asignaciones encadenadas (aunque Patito no las soporta, sigue siendo la respuesta semánticamente correcta).

El return Tipo::ERROR final (línea 70) es inalcanzable en teoría — el switch cubre todos los operadores del enum — pero ahí está como red de seguridad por si alguien agrega un operador nuevo y olvida un case. Sin él, algunos compiladores se quejarían de "control reaches end of non-void function".

VariableInfo y ParametroInfo

31struct VariableInfo { 32 std::string nombre; 33 Tipo tipo; 34 int linea; // linea donde se declaro (para mensajes de error) 35};

Estructura plana con los tres datos por variable: nombre, tipo y línea de declaración. Uso struct (no class) porque todos los campos son públicos y no necesito invariantes. La línea se guarda solo para mensajes de error legibles (p. ej. "x ya estaba declarada en línea 12").

Más adelante (Etapa 3) este struct crecerá con una dirección de memoria virtual, pero por ahora 3 campos bastan.

VariableTable

37class VariableTable { 38public: 39 // Inserta una variable. Devuelve false si ya existia (duplicado). 40 bool agregar(const std::string& nombre, Tipo tipo, int linea); 41 42 bool existe(const std::string& nombre) const; 43 const VariableInfo* obtener(const std::string& nombre) const; 44 std::size_t tamano() const { return tabla.size(); } 45 46 // Iteracion (lectura). 47 auto begin() const { return tabla.begin(); } 48 auto end() const { return tabla.end(); } 49 50private: 51 std::unordered_map<std::string, VariableInfo> tabla; 52};

Wrapper sobre std::unordered_map con una API minimal:

  • agregar — inserta y devuelve false si el nombre ya estaba. Esta semántica permite que el llamador detecte duplicados con un solo if, sin tener que llamar primero a existe (que sería una doble búsqueda).
  • existe y obtener — para usos futuros (Etapa 3 los necesita para validar "variable usada sin declarar").
  • tamano, begin, end — para el dump del --dir.

Por qué wrapper y no usar el map directamente: en el futuro puedo agregar invariantes (por ejemplo, "no se puede agregar después de cerrar el scope") sin tocar a los llamadores. También oculta el tipo concreto del map: si decido cambiar a std::map o a una hash map propia, solo cambia este header.

54struct ParametroInfo { 55 std::string nombre; 56 Tipo tipo; 57};

Datos por parámetro. No tiene linea porque al imprimir no necesito esa info (los parámetros se reportan junto con su función). Se mantiene como struct separado de VariableInfo para que la lista ordenada de parámetros pueda evolucionar independientemente (p. ej., agregar "pasaje por referencia" más adelante).

FunctionInfo

59struct FunctionInfo { 60 std::string nombre; 61 Tipo tipoRetorno; 62 std::vector<ParametroInfo> parametros; // ordenados 63 VariableTable variables; // tabla local de la funcion 64 int linea; 65};

Toda la información asociada a una función:

  • nombre y tipoRetorno — auto-explicativos.
  • parametros — vector, no VariableTable. El orden importa para validar llamadas en Etapa 3 (los argumentos se pasan posicionalmente). Si los guardara en una hash map perdería el orden.
  • variables — la tabla local. Cada parámetro también se inserta aquí para que el detector de duplicados los considere al mismo nivel que las vars locales: en Patito, nula f(x: entero) { vars x : entero; ... } es un duplicado, y este modelo lo atrapa naturalmente.
  • linea — dónde se declaró, para ordenar el dump determinísticamente.

FunctionDirectory

67class FunctionDirectory { 68public: 69 // Registra una nueva funcion. Devuelve nullptr si el nombre ya estaba en uso. 70 // Si la insercion tiene exito, devuelve un puntero estable a la entrada 71 // creada (el unordered_map garantiza estabilidad de punteros a value). 72 FunctionInfo* agregar(const std::string& nombre, Tipo tipoRetorno, int linea); 73 74 bool existe(const std::string& nombre) const; 75 FunctionInfo* obtener(const std::string& nombre); 76 const FunctionInfo* obtener(const std::string& nombre) const; 77 78 std::size_t tamano() const { return directorio.size(); } 79 80 auto begin() const { return directorio.begin(); } 81 auto end() const { return directorio.end(); } 82 83 // Dump legible del directorio (para depuracion / bandera --dir). 84 void imprimir(std::ostream& os) const; 85 86 private: 87 std::unordered_map<std::string, FunctionInfo> directorio; 88}; 89 90} // namespace patito

API similar a VariableTable pero con dos peculiaridades:

  • agregar devuelve un puntero al FunctionInfo recién creado (o nullptr si hubo duplicado). Esto le ahorra al llamador una segunda búsqueda: tras un agregar exitoso, puede empezar a llenar parametros y variables directamente sobre el puntero. std::unordered_map garantiza que sus referencias/punteros a valores no se invalidan cuando se insertan o borran otras entradas (a diferencia de std::vector).
  • Hay dos overloads de obtener: uno const y uno mutable. El const sirve para el dump (que no debe modificar); el mutable sirve para que el analizador agregue variables locales al scope activo.

imprimir es la operación que la bandera --dir del binario invoca para volcar todo el directorio al stdout.

Implementación de VariableTable

11bool VariableTable::agregar(const std::string& nombre, Tipo tipo, int linea) { 12 if (tabla.count(nombre)) return false; // duplicado 13 tabla.emplace(nombre, VariableInfo{nombre, tipo, linea}); 14 return true; 15}

Inserción con detección de duplicado:

  • Línea 12: count(nombre) devuelve 0 o 1 (es un set/map). Si ya existe, regresa false sin modificar nada.
  • Línea 13: emplace construye el VariableInfo en su lugar dentro del map (evita una copia adicional). El struct se inicializa con sintaxis de llaves ({nombre, tipo, linea}).
17bool VariableTable::existe(const std::string& nombre) const { 18 return tabla.count(nombre) > 0; 19} 20 21const VariableInfo* VariableTable::obtener(const std::string& nombre) const { 22 auto it = tabla.find(nombre); 23 return it == tabla.end() ? nullptr : &it->second; 24}

Dos consultas estándar sobre el map. obtener devuelve puntero (no referencia) para representar "no encontrado" con nullptr, evitando excepciones.

Implementación de FunctionDirectory

28FunctionInfo* FunctionDirectory::agregar(const std::string& nombre, 29 Tipo tipoRetorno, int linea) { 30 if (directorio.count(nombre)) return nullptr; // duplicado 31 auto [it, ok] = directorio.emplace( 32 nombre, 33 FunctionInfo{nombre, tipoRetorno, {}, {}, linea}); 34 return ok ? &it->second : nullptr; 35}

Misma lógica que VariableTable::agregar pero devuelve puntero estable a la entrada nueva:

  • Línea 30: si ya existe, nullptr.
  • Líneas 31-33: emplace retorna un pair<iterator, bool>. La sintaxis auto [it, ok] = ... es structured bindings (C++17) que desempaqueta el par en dos variables.
  • El FunctionInfo{nombre, tipoRetorno, {}, {}, linea} se construye con dos {} vacíos para inicializar el vector de parámetros y la VariableTable sin elementos.
  • Línea 34: aunque ya verifiqué que no existe, devuelvo nullptr defensivamente si por alguna razón emplace falló (no debería pasar).

Tres consultas. Dos versiones de obtener (const y mutable) porque a veces necesito modificar la entrada (al agregar vars locales) y otras solo leerla (al imprimir). El compilador escoge automáticamente la versión correcta según si la instancia es const o no.

imprimir() con orden determinista

51void FunctionDirectory::imprimir(std::ostream& os) const { 52 os << "=== DIRECTORIO DE FUNCIONES (" << directorio.size() << ") ===\n";

Encabezado del dump. Imprime el conteo total, útil para verificar a simple vista cuántos scopes hay.

54 // Ordenamos por linea de declaracion para que la salida sea determinista 55 // y siga el orden natural en que aparecen en el archivo fuente (no el 56 // orden interno del unordered_map, que depende del hash). 57 std::vector<const FunctionInfo*> ordenadas; 58 ordenadas.reserve(directorio.size()); 59 for (const auto& [nombre, info] : directorio) { 60 ordenadas.push_back(&info); 61 } 62 std::sort(ordenadas.begin(), ordenadas.end(), 63 [](const FunctionInfo* a, const FunctionInfo* b) { 64 return a->linea < b->linea; 65 });

El problema que esto resuelve: std::unordered_map no garantiza orden al iterar. Dos corridas pueden devolver las funciones en orden distinto (depende del hash de las strings y del bucket layout). Eso haría que el output de --dir cambie entre corridas — un revisor podría pensar "el doc dice que aparecen en orden X pero yo veo Y; ¿está roto?".

La solución: copiar punteros a un vector y ordenarlos por linea antes de imprimir. El resultado es que las funciones siempre aparecen en el orden en que se declararon en el archivo fuente.

  • Línea 57-58: vector de punteros (no copias, para no duplicar memoria). reserve evita reasignaciones durante el push_back.
  • Línea 59-61: copio los punteros. La sintaxis auto& [nombre, info] es structured bindings sobre los pares del map.
  • Línea 62-65: std::sort con un lambda como comparador. El lambda recibe dos punteros y devuelve true si a va antes que b.
67 for (const FunctionInfo* infoPtr : ordenadas) { 68 const FunctionInfo& info = *infoPtr; 69 os << " funcion '" << info.nombre << "' retorno=" << tipoToStr(info.tipoRetorno) 70 << " linea=" << info.linea << "\n";

Itero el vector ya ordenado y imprimo el encabezado de cada función: nombre, tipo de retorno (convertido a texto con tipoToStr) y línea de declaración. La línea 68 desreferencia el puntero para tener una referencia conveniente.

71 if (!info.parametros.empty()) { 72 os << " parametros:"; 73 for (const auto& p : info.parametros) { 74 os << " " << p.nombre << ":" << tipoToStr(p.tipo); 75 } 76 os << "\n"; 77 }

Si la función tiene parámetros, los lista en orden (vienen ya ordenados en el vector). Formato: x:entero y:flotante z:entero. No necesito ordenar este vector porque YO lo construí en orden al procesar la declaración.

Bandera que vale lo que dice el comentario. El problema concreto: ANTLR siempre recorre el subárbol completo, incluso si decidimos "ignorar" la función entera. La bandera evita ese efecto: una vez activada, enterDeclaracion se vuelve no-op hasta que exitFuncs la apague.

Este es el primer override y dispara una vez por programa (la regla raíz).

  • Línea 43: getSymbol()->getLine() da la línea del token. static_cast<int> convierte el size_t que devuelve ANTLR a int (que es lo que mi struct espera).
  • Línea 45: registra al programa en el directorio como una "función" de tipo retorno NULA. Esta es la decisión de diseño clave: el scope global vive en el mismo contenedor que las funciones del usuario, lo cual simplifica el resto del código.
