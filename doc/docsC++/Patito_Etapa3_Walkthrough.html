Patito Etapa 3 — Walkthrough línea por línea

Generación de cuádruplos · Memoria virtual · Pilas + Fila · Luis Manuel Gonzalez Martinez · A01722501

Visión general

Etapa 3 agrega generación de código intermedio sobre lo que ya tenía Etapa 2. El analizador semántico ahora, además de llenar tablas, emite una secuencia de cuádruplos (operaciones de la forma (op, izq, der, resultado)) que representan el programa en una forma más cercana a lo que la VM (Etapa 5) podrá ejecutar.

Pipeline actualizado:
archivo.patito → Lexer → Parser → SemanticAnalyzer (listener) → Tabla de Constantes + Fila de Cuádruplos

Lo nuevo en esta etapa son 7 archivos nuevos y modificaciones a 5 archivos existentes. El walkthrough cubre los archivos nuevos en su totalidad y muestra solo las partes añadidas/cambiadas de los modificados.

  • Memoria virtual: cada operando recibe una dirección entera dentro de uno de 12 rangos predefinidos. Esto lo maneja AsignadorMemoria.
  • Deduplicación de constantes: los literales iguales comparten dirección. Lo maneja TablaConstantes.
  • Las 3 PILAS + la FILA: conviven dentro de GeneradorCuadruplos, con la lógica para emitir cuádruplos binarios, asignaciones y prints.
  • 9 puntos neurálgicos nuevos en el SemanticAnalyzer que reaccionan al árbol de parseo y producen los cuádruplos.

src/semantic/Cuadruplo.h

Nuevo
src/semantic/Cuadruplo.h
Struct que representa una operación de código intermedio. Es la unidad que el generador emite y que la VM (Etapa 5) consumirá.
1// Cuadruplo: unidad de codigo intermedio que produce el compilador. 2// Cada cuadruplo es una operacion de la forma: 3// 4// (operador, operando_izq, operando_der, resultado) 5// 6// Donde operandos y resultado son DIRECCIONES de memoria virtual.

Comentario de cabecera. Explica el formato de un cuádruplo y, crucial, que los campos son direcciones, no nombres ni valores. La VM en Etapa 5 traducirá esas direcciones a celdas reales de memoria.

21struct Cuadruplo { 22 Operador op; 23 int opIzq; // direccion, o -1 si no aplica 24 int opDer; // direccion, o -1 si no aplica 25 int resultado; // direccion del destino 26};

Struct plano con 4 campos. Uso int para las direcciones porque los rangos definidos (1000–15999) caben de sobra. El sentinel -1 indica "no aplica" — útil para operaciones unarias o de un solo operando (asignación, print).

  • Binario (+ - * / etc.): los 3 campos llenos.
  • Asignación (=): opIzq = dirección origen, opDer = -1, resultado = destino.
  • PRINT: opIzq = -1, opDer = -1, resultado = dirección del valor a imprimir.

src/semantic/AsignadorMemoria

Nuevo
src/semantic/AsignadorMemoria.h + .cpp
Reparte direcciones virtuales de los 9 rangos no-constantes (globales, locales, temporales × 3 tipos cada uno).

Header — definición de rangos

26namespace Rangos { 27 constexpr int GLOBAL_ENT_INI = 1000; 28 constexpr int GLOBAL_ENT_FIN = 1999; 29 constexpr int GLOBAL_FLOT_INI = 2000; 30 constexpr int GLOBAL_FLOT_FIN = 2999; 31 constexpr int GLOBAL_BOOL_INI = 3000; 32 constexpr int GLOBAL_BOOL_FIN = 3999; 33 34 constexpr int LOCAL_ENT_INI = 5000; 35 constexpr int LOCAL_ENT_FIN = 5999; 36 ... 49 constexpr int CTE_LETRERO_INI = 15000; 50 constexpr int CTE_LETRERO_FIN = 15999; 51}

Los 12 rangos definidos como constexpr int dentro de un namespace anidado. Hacerlos constexpr permite que el compilador los evalúe en tiempo de compilación y los use en condiciones del propio compilador.

Por qué huecos en 4000s, 8000s, 12000s: el primer dígito (o dos primeros) de cualquier dirección revela inmediatamente el segmento + tipo al debuggear. Esa es la razón estética/práctica del diseño.

La VM (Etapa 5+) importará este namespace para saber a qué arreglo dirigirse cuando lea un operando: if (addr < GLOBAL_FLOT_INI) usarEnterosGlobales[addr - 1000];

53class AsignadorMemoria { 54public: 55 AsignadorMemoria(); 56 57 // Resetea contadores locales y temporales al entrar a una nueva funcion. 58 // Globales NO se tocan. 59 void resetScopeLocal(); 60 61 int nuevaGlobal(Tipo tipo); 62 int nuevaLocal(Tipo tipo); 63 int nuevoTemporal(Tipo tipo); 64 65private: 66 int gEnt, gFlot, gBool; // globales 67 int lEnt, lFlot, lBool; // locales (reset por funcion) 68 int tEnt, tFlot, tBool; // temporales (reset por funcion) 69};

API minimal con 4 métodos. 9 contadores enteros como estado, uno por (segmento × tipo) no-constante. Las constantes no están aquí — viven en TablaConstantes porque requieren deduplicación (un map, no un contador).

resetScopeLocal() es la clave del esquema por activation records: cada función arranca sus locales y temporales en el mismo número (5000, 6000, 9000, etc.) sin chocar con otras funciones porque la VM les dará espacio fresco al hacer la llamada.

Implementación

5AsignadorMemoria::AsignadorMemoria() 6 : gEnt(Rangos::GLOBAL_ENT_INI), 7 gFlot(Rangos::GLOBAL_FLOT_INI), 8 gBool(Rangos::GLOBAL_BOOL_INI), 9 lEnt(Rangos::LOCAL_ENT_INI), 10 lFlot(Rangos::LOCAL_FLOT_INI), 11 lBool(Rangos::LOCAL_BOOL_INI), 12 tEnt(Rangos::TEMP_ENT_INI), 13 tFlot(Rangos::TEMP_FLOT_INI), 14 tBool(Rangos::TEMP_BOOL_INI) {}

Constructor: inicializa cada contador con el inicio de su rango. Lista de inicializadores es la forma idiomática de C++ para esto (más eficiente que asignaciones dentro del cuerpo).

16void AsignadorMemoria::resetScopeLocal() { 17 lEnt = Rangos::LOCAL_ENT_INI; 18 lFlot = Rangos::LOCAL_FLOT_INI; 19 lBool = Rangos::LOCAL_BOOL_INI; 20 tEnt = Rangos::TEMP_ENT_INI; 21 tFlot = Rangos::TEMP_FLOT_INI; 22 tBool = Rangos::TEMP_BOOL_INI; 23}

Resetea 6 contadores (locales + temporales × 3 tipos) al inicio de su rango. Globales no se tocan porque persisten entre funciones.

El SemanticAnalyzer llamará a esto en enterFuncs, antes de procesar los parámetros de la nueva función.

25int AsignadorMemoria::nuevaGlobal(Tipo tipo) { 26 switch (tipo) { 27 case Tipo::ENTERO: 28 if (gEnt > Rangos::GLOBAL_ENT_FIN) return -1; 29 return gEnt++; 30 case Tipo::FLOTANTE: 31 if (gFlot > Rangos::GLOBAL_FLOT_FIN) return -1; 32 return gFlot++; 33 case Tipo::BOOL: 34 if (gBool > Rangos::GLOBAL_BOOL_FIN) return -1; 35 return gBool++; 36 default: 37 return -1; // NULA, ERROR no son asignables como var 38 } 39}

Patrón repetido para los 3 métodos (Global / Local / Temporal). Para cada tipo: verifica que no se haya pasado el fin del rango, devuelve el contador y lo incrementa (post-incremento) en una sola línea.

El sentinel -1 de overflow lo detecta el analizador, que reporta error. En programas pedagógicos no se llega ni de cerca al límite (1000 variables enteras globales).

src/semantic/TablaConstantes

Nuevo
src/semantic/TablaConstantes.h + .cpp
Asigna dirección virtual única a cada literal (entero, flotante, letrero). Si el mismo literal aparece dos veces, devuelve la misma dirección (deduplicación).

Header

21class TablaConstantes { 22public: 23 TablaConstantes(); 24 25 // Devuelve la direccion del literal numerico. Si ya existia, devuelve 26 // la misma; si no, asigna una nueva del rango del tipo correspondiente. 27 // Solo acepta tipo ENTERO o FLOTANTE; otros devuelven -1. 28 int direccionDe(const std::string& lexema, Tipo tipo); 29 30 // Version separada para letreros (cadenas entre comillas), porque 31 // el enum Tipo no incluye un valor "letrero" (es de uso interno). 32 int direccionDeLetrero(const std::string& lexema); 33 34 void imprimir(std::ostream& os) const; 35 36 std::size_t tamano() const { return ents.size() + flots.size() + letreros.size(); } 37 38private: 39 std::unordered_map<std::string, int> ents; // "5" -> 13000 40 std::unordered_map<std::string, int> flots; // "3.14" -> 14000 41 std::unordered_map<std::string, int> letreros; // "\"hola\"" -> 15000 42 43 int contEnt, contFlot, contLet; 44};

Tres hash maps (uno por tipo de constante). La clave es el lexema literal tal como apareció en el código fuente: "5", "3.14", "\"hola\"" (las comillas son parte del lexema del letrero).

Dos métodos en lugar de uno con overload: direccionDe(lex, tipo) para numéricos y direccionDeLetrero(lex) para letreros. Esta separación apareció en una refactorización temprana: el primer intento usaba un único método polimórfico pasando un Tipo "dummy" para letreros, lo cual era confuso porque el enum Tipo no tiene un valor "LETRERO".

Implementación con deduplicación

15int TablaConstantes::direccionDe(const std::string& lexema, Tipo tipo) { 16 switch (tipo) { 17 case Tipo::ENTERO: { 18 auto it = ents.find(lexema); 19 if (it != ents.end()) return it->second; // dedup hit 20 if (contEnt > Rangos::CTE_ENT_FIN) return -1; 21 int dir = contEnt++; 22 ents.emplace(lexema, dir); 23 return dir; 24 } 25 ...

El patrón de deduplicación: busca primero, si encuentra devuelve la dirección existente; si no, asigna nueva y la registra.

Por eso un programa que dice escribe("hola"); escribe("mundo"); escribe("hola"); solo registra "hola" una vez (dirección 15000). El segundo "hola" devuelve 15000 de nuevo. Esto se verifica en el test 11_escribe_letrero.patito.

51void TablaConstantes::imprimir(std::ostream& os) const { 52 os << "=== TABLA DE CONSTANTES (" << tamano() << ") ===\n"; 53 54 auto dump = [&os](const std::unordered_map<std::string, int>& m, 55 const char* etiqueta) { 56 if (m.empty()) return; 57 std::vector<std::pair<std::string, int>> ordenadas(m.begin(), m.end()); 58 std::sort(ordenadas.begin(), ordenadas.end(), 59 [](const auto& a, const auto& b) { return a.second < b.second; }); 60 ...

Mismo problema que en FunctionDirectory::imprimir: unordered_map no garantiza orden. Solución: copiar a vector<pair> y ordenar por dirección antes de imprimir.

El lambda dump es una utility local para evitar repetir el código tres veces (una por map).

src/semantic/GeneradorCuadruplos

Nuevo
src/semantic/GeneradorCuadruplos.h + .cpp
Las 3 PILAS + la FILA + las operaciones de generación. Es el corazón de la Etapa 3.

Header — las 3 pilas

58private: 59 std::stack<int> pilaOperandos; 60 std::stack<Tipo> pilaTipos; 61 std::stack<Operador> pilaOperadores; 62 std::vector<Cuadruplo> _fila; 63 AsignadorMemoria& mem; 64 TablaConstantes& cte;

Las 3 pilas (las que pide la rúbrica) son std::stack<T> — wrappers sobre std::deque con la API estándar de pila (push, pop, top).

  • pilaOperandos: direcciones de operandos pendientes (es lo que apilan exitFactor cuando ve un id o cte).
  • pilaTipos: en paralelo con pilaOperandos. Mismo tamaño siempre. Cada vez que algo entra a una, también entra a la otra.
  • pilaOperadores: operadores empujados al entrar a una regla binaria (enterExp/enterTermino) y popeados al salir.

La fila es un std::vector<Cuadruplo> en lugar de std::queue porque Etapa 4 necesitará acceder por índice (backpatching).

Las referencias mem y cte permiten al generador asignar temporales y registrar constantes sin recibirlos en cada llamada.

Implementación — popYEmitirBinario

22Tipo GeneradorCuadruplos::popYEmitirBinario() { 23 _ultimoPropagado = false; 24 25 if (pilaOperadores.empty() || pilaOperandos.size() < 2) { 26 return Tipo::ERROR; 27 } 28 29 Operador op = pilaOperadores.top(); pilaOperadores.pop(); 30 31 int dDer = pilaOperandos.top(); pilaOperandos.pop(); 32 Tipo tDer = pilaTipos.top(); pilaTipos.pop(); 33 34 int dIzq = pilaOperandos.top(); pilaOperandos.pop(); 35 Tipo tIzq = pilaTipos.top(); pilaTipos.pop(); 36 37 // Si alguno de los operandos ya viene marcado como ERROR (p.ej. var 38 // no declarada reportada en exitFactor), propagamos en silencio. 39 if (tIzq == Tipo::ERROR || tDer == Tipo::ERROR) { 40 _ultimoPropagado = true; 41 pilaOperandos.push(-1); 42 pilaTipos.push(Tipo::ERROR); 43 return Tipo::ERROR; 44 } 45 46 Tipo tRes = SemanticCube::resultado(tIzq, op, tDer); 47 if (tRes == Tipo::ERROR) { 48 pilaOperandos.push(-1); 49 pilaTipos.push(Tipo::ERROR); 50 return Tipo::ERROR; 51 } 52 53 int dRes = mem.nuevoTemporal(tRes); 54 _fila.push_back(Cuadruplo{op, dIzq, dDer, dRes}); 55 56 pilaOperandos.push(dRes); 57 pilaTipos.push(tRes); 58 59 return tRes; 60}

Esta es la operación central de la etapa. Cada operación binaria del programa (cada +, -, *, /, <, etc.) termina aquí. Pasos:

  1. Línea 23: limpia el flag _ultimoPropagado. El caller lo lee después para decidir si suprimir el reporte (ver "Propagación de ERROR" más abajo).
  2. Línea 25: defensa contra pilas mal balanceadas (no debería pasar si los enters/exits están bien).
  3. Línea 29: pop del operador (LIFO: el último que entró sale primero).
  4. Líneas 31-32: pop del operando DERECHO. Importante: viene primero en la pila porque fue el último empujado.
  5. Líneas 34-35: pop del operando IZQUIERDO.
  6. Líneas 39-44: propagación silenciosa. Si uno de los operandos ya es ERROR (porque exitFactor reportó "variable no declarada"), no consultamos el cubo: empujamos un sentinel ERROR de regreso a las pilas y marcamos _ultimoPropagado. Así el caller suprime el reporte redundante de "tipos incompatibles" y solo ve el error raíz una vez.
  7. Línea 46: consulta al cubo semántico — la validación de tipos finalmente se usa aquí, después de haberse diseñado en Etapa 2.
  8. Líneas 47-51: si el cubo dijo ERROR (mismatch genuino), empuja el sentinel ERROR a las pilas y devuelve. El analizador reporta el mensaje porque _ultimoPropagado es false.
  9. Línea 53: pedir un temporal del tipo del resultado.
  10. Línea 54: emitir el cuádruplo: (op, izq, der, temporal).
  11. Líneas 56-57: push del resultado a las pilas para que la operación enclosing lo use como operando.

Por qué push de ERROR en lugar de no hacer nada: mantener el balance de las pilas. Si esta operación falla pero la pila queda con dos operandos menos y ningún resultado empujado, la operación enclosing hace underflow.

62Tipo GeneradorCuadruplos::emitirAsignacion(int direccionDestino, Tipo tipoDestino) { 63 _ultimoPropagado = false; 64 if (pilaOperandos.empty()) return Tipo::ERROR; 65 66 int dOrigen = pilaOperandos.top(); pilaOperandos.pop(); 67 Tipo tOrigen = pilaTipos.top(); pilaTipos.pop(); 68 69 if (tOrigen == Tipo::ERROR) { 70 _ultimoPropagado = true; 71 return Tipo::ERROR; 72 } 73 74 Tipo tRes = SemanticCube::resultado(tipoDestino, Operador::ASIGNA, tOrigen); 75 if (tRes == Tipo::ERROR) { 76 return Tipo::ERROR; 77 } 78 79 _fila.push_back(Cuadruplo{Operador::ASIGNA, dOrigen, -1, direccionDestino}); 80 return tRes; 81}

Asignación: similar a binario pero asimétrico. El destino se pasa como parámetro (lo resuelve el analizador desde el lado izquierdo del =), el origen se pop de la pila.

Líneas 69-72: misma idea que popYEmitirBinario — si el RHS ya era ERROR (p.ej. x = y; con y no declarada), propagamos en silencio en lugar de reportar "tipos incompatibles" además del error de variable no declarada.

El cubo valida que (tipoDestino, =, tipoOrigen) sea legal. Esto detecta cosas como x = 3.14; donde x es entero (cubo devuelve ERROR para esa combinación).

El cuádruplo emitido es (=, origen, -1, destino). El -1 indica que el campo opDer no aplica.

62void GeneradorCuadruplos::emitirPrint(int direccion) { 63 _fila.push_back(Cuadruplo{Operador::PRINT, -1, -1, direccion}); 64}

El más simple. Solo recibe una dirección y emite un PRINT. Lo llama el analizador en exitImpElem una vez por elemento del escribe(...).

66void GeneradorCuadruplos::emitirDirecto(const Cuadruplo& q) { 67 _fila.push_back(q); 68}

Vía de escape para casos especiales como el signo unario, donde el cuádruplo no surge del flujo normal de pilas. Lo usa exitFactor para emitir (-, 0, var, t) cuando ve -id.

Volcado legible para la bandera --cuadruplos. Cada cuádruplo en una línea, con índice (sirve para Etapa 4 donde se referenciarán por índice en GOTOs). Los campos -1 se muestran como _ para no ensuciar el output con valores especiales.

Helpers nuevos

const VariableInfo* SemanticAnalyzer::buscarVar(const std::string& nombre) const { // Primero scope local actual, luego global. if (scopeActual) { if (auto* v = scopeActual->variables.obtener(nombre)) return v; } if (scopeActual && scopeActual->nombre != nombrePrograma) { auto* global = const_cast<FunctionDirectory&>(funcs).obtener(nombrePrograma); if (global) { if (auto* v = global->variables.obtener(nombre)) return v; } } return nullptr; }

PN3-A: exitFactor

void SemanticAnalyzer::exitFactor(PatitoParser::FactorContext* ctx) { if (enFuncDuplicada) return; // Caso 1: ( expresion ) if (ctx->PIZQ() != nullptr) { return; // ya hay un operando en el top de pilaOperandos } // Caso 2: llamada if (ctx->llamada() != nullptr) { ... int dir = mem.nuevoTemporal(f->tipoRetorno); gen.pushOperando(dir, f->tipoRetorno); return; } // Casos 3 y 4: signoOpc ID o signoOpc cte ... }

El factor tiene 4 alternativas en la gramática, y cada una se maneja distinto:

  1. ( expresion ): no hacer nada. La expresión interna ya ejecutó su propio flujo de exitExp/exitTermino y dejó su resultado en el top de pilaOperandos.
  2. llamada: reservar un temporal del tipo de retorno y empujarlo como "fantasma". Etapa 4 emitirá los cuádruplos reales (ERA/PARAM/GOSUB) que llenarán ese temporal con el verdadero retorno.
  3. signoOpc ID: buscar la variable, manejar signo unario, empujar.
  4. signoOpc cte: registrar la constante (con signo si aplica), empujar.
if (ctx->ID() != nullptr) { // signoOpc ID ... const VariableInfo* v = buscarVar(nombre); if (v == nullptr) { error(l, c, "variable '" + nombre + "' usada sin declarar en '" + (scopeActual ? scopeActual->nombre : "?") + "'"); gen.pushOperando(-1, Tipo::ERROR); return; } if (tieneMenos) { // Generamos t = 0 - id (signo unario aplicado a variable). int cero = cte.direccionDe("0", Tipo::ENTERO); int t = mem.nuevoTemporal(v->tipo); gen.emitirDirecto(Cuadruplo{Operador::MENOS, cero, v->direccion, t}); gen.pushOperando(t, v->tipo); } else { gen.pushOperando(v->direccion, v->tipo); } return; }

Caso ID con posible signo:

  • Busca la variable. Si no existe, reporta error y empuja (-1, ERROR) como phantom para no romper operaciones encerradoras.
  • Si tiene signo -, emite manualmente el cuádruplo de negación t = 0 - id (donde 0 es una constante registrada en la tabla). Esto requiere el método emitirDirecto porque la operación no fluye por las pilas normalmente.
  • Si no hay signo (o es +), simplemente empuja la dirección y tipo de la variable.
if (ctx->cte() != nullptr) { // signoOpc cte ... if (tieneMenos) { lex = "-" + lex; // tratamos "-5" como una constante distinta de "5" } int dir = cte.direccionDe(lex, t); ... gen.pushOperando(dir, t); return; }

Caso cte con posible signo: estrategia distinta — en lugar de generar un cuádruplo de negación, tratamos -5 como una constante distinta de 5. La tabla de constantes registra ambos lexemas por separado.

Ventaja: ahorra un cuádruplo en el caso común de literales negativos.

PN3-B: enterExp y exitExp

Patrón gemelo (enter + exit) para operaciones aritméticas de suma/resta.

  • enterExp: si la regla incluye un operador (MAS o MENOS), lo empuja a pilaOperadores. La detección se basa en si el ctx tiene el TerminalNode correspondiente.
  • exitExp: si había operador, llama a popYEmitirBinario. Si el cubo dio ERROR y no fue una propagación silenciosa (ultimoFuePropagado() false), reporta con la posición del operador. Esto evita duplicar mensajes cuando un operando ya venía marcado como ERROR desde más abajo.

El caso "exp es solo un termino" (sin operador) cae en el early-return en ambas funciones — no hace nada, deja el flujo del termino tal cual.

PN3-C: enterTermino y exitTermino

Análogo a exp pero con multiplicación y división. La estructura es idéntica — solo cambian los operadores que detecta.

La precedencia entre */ y +- sale "gratis" del árbol: como termino vive dentro de exp en la gramática, exitTermino siempre dispara antes que el exitExp que lo contiene, garantizando que a + b * c emite primero b * c y luego a + (resultado).

Los operadores relacionales viven en una sub-regla aparte (opRel) porque la gramática los separa así. enterRelOpc detecta cuál de los 4 está presente y lo empuja.

El pop+emit ocurre en exitExpresion (no exitRelOpc) porque la expresión completa a < b tiene su rama derecha (b) parseada DESPUÉS de relOpc. Solo cuando todo eso terminó es seguro hacer el pop.

El resultado de cualquier relacional es siempre tipo BOOL (el cubo lo devuelve), que se asigna a un temporal en el rango 11000–11999.

PN3-E: exitAsigna

void SemanticAnalyzer::exitAsigna(PatitoParser::AsignaContext* ctx) { if (enFuncDuplicada) return; auto* idNode = ctx->ID(); std::string nombre = idNode->getText(); ... const VariableInfo* v = buscarVar(nombre); if (v == nullptr) { error(...); if (!gen.pilasOperandosVacia()) gen.popOperando(); return; } Tipo res = gen.emitirAsignacion(v->direccion, v->tipo); if (res == Tipo::ERROR && !gen.ultimoFuePropagado()) { error(l, c, "tipos incompatibles en asignacion a '" + nombre + "' (" + tipoToStr(v->tipo) + ")"); } }

Punto neurálgico de la asignación. Pasos:

  1. Toma el id del lado izquierdo del =.
  2. Lo busca en scope. Si no existe, reporta error y limpia el operando del RHS para que no contamine la siguiente expresión.
  3. Si existe, delega a gen.emitirAsignacion con la dirección y tipo del destino.
  4. Si el cubo rechaza la combinación (asignar flotante a entero) y no fue una propagación silenciosa, reporta. La cláusula !gen.ultimoFuePropagado() evita duplicar mensajes cuando el RHS ya venía como ERROR (p.ej. x = y; con y no declarada — se reporta una vez en exitFactor, no dos veces).

PN3-F: exitImpElem

void SemanticAnalyzer::exitImpElem(PatitoParser::ImpElemContext* ctx) { if (enFuncDuplicada) return; if (ctx->LETRERO() != nullptr) { std::string lex = ctx->LETRERO()->getText(); int dir = cte.direccionDeLetrero(lex); ... gen.emitirPrint(dir); return; } // Era una expresion: el top de pilaOperandos es lo que hay que imprimir. if (gen.pilasOperandosVacia()) return; int dir = gen.topOperando(); gen.emitirPrint(dir); gen.popOperando(); }

Cada elemento del escribe(...) dispara este punto:

  • Si es un letrero: registrar en la tabla (con deduplicación) y emitir un PRINT con esa dirección.
  • Si es una expresión: ya está su resultado en el top de pilaOperandos. Pop, emitir PRINT, listo.

Por eso un escribe("hola", x, a + b) genera 3 cuádruplos PRINT (más los cuádruplos para evaluar a + b que se emitieron antes).

enterFuncs ahora hace dos cosas más allá de su trabajo de Etapa 2:

  1. Llama mem.resetScopeLocal() apenas se confirma que estamos entrando a una función válida. Esto pone los contadores locales (5000+) y temporales (9000+) en su valor inicial para esta función.
  2. Por cada parámetro, asigna una dirección local con mem.nuevaLocal(ptipo) antes de registrarlo en la tabla.
void SemanticAnalyzer::enterDeclaracion(PatitoParser::DeclaracionContext* ctx) { ... bool esGlobal = (scopeActual->nombre == nombrePrograma); auto procesarId = [&](antlr4::tree::TerminalNode* idNode) { ... int direccion = esGlobal ? mem.nuevaGlobal(tipo) : mem.nuevaLocal(tipo); ... scopeActual->variables.agregar(nombre, tipo, l, direccion); }; ... }

enterDeclaracion decide si la variable va al rango global o al rango local según el scope actual. Esto es lo que permite que x declarada en el programa principal vaya a 1000 y x declarada en una función vaya a 5000.
