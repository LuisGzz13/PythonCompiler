4.1 Imports + defaults + rangos

from semantico import AsignadorMemoria
from cuadruplos import TablaConstantes

_DEFAULTS = {
    "entero":   0,
    "flotante": 0.0,
    "bool":     0,    # bool en patito es 0 o 1 (no Python True/False)
}

_RANGOS = []
for (seg, tipo), base in AsignadorMemoria._BASE.items():
    _RANGOS.append((base, base + 1000, seg, tipo))
for tipo, base in TablaConstantes._BASE.items():
    _RANGOS.append((base, base + 1000, "cte", tipo))

¿Por qué _RANGOS se construye al cargar el módulo? Para que la VM siempre esté sincronizada con los rangos del compilador. Si AsignadorMemoria._BASE cambia (por ejemplo, agregar el segmento retorno), la VM automáticamente lo refleja sin tocar código. Única fuente de verdad.

4.2 _segmento_y_tipo(direccion)

def _segmento_y_tipo(direccion):
    """Decodifica una direccion virtual en (segmento, tipo). None si invalida."""
    if direccion is None:
        return None
    for start, end, seg, tipo in _RANGOS:
        if start <= direccion < end:
            return (seg, tipo)
    return None

Recorre los 13 rangos derivados de AsignadorMemoria._BASE + TablaConstantes._BASE. Devuelve ("global", "entero") para 1500, ("retorno", "flotante") para 18002, etc.

4.3 La clase Frame

class Frame:
    """Un frame de ejecucion: memoria local + temporal de UNA funcion activa."""
    def __init__(self, nombre):
        self.nombre = nombre      # debug y trazas
        self.locales = {}         # {direccion -> valor} para 5000s/6000s/7000s
        self.temps = {}           # {direccion -> valor} para 9000s/10000s/11000s
        self.ip_retorno = None    # IP al que volver al ENDFUNC

Cada llamada a una función crea un Frame nuevo. Las locales y temps de ese frame son AISLADAS — cada frame parte sus 5000s/9000s en cero. Esto permite que dos funciones distintas usen la misma dirección local 5000 sin chocar (porque cada una vive en su propio Frame).

4.4 MaquinaVirtual.__init__

def __init__(self, analizador):
    self.fila = analizador.gen.fila
    self.func_dir = analizador.func_dir
    self.nombre_programa = analizador.nombre_programa

    # Memoria global: persiste durante TODA la ejecucion
    self.memoria_global = {}

    # Memoria de constantes: direccion -> valor (con type-conversion)
    self.memoria_constantes = self._cargar_constantes(analizador.cte)

    # Pila de frames de ejecucion. Main es el bottom (frame[0]).
    main = self._nuevo_frame(self.nombre_programa)
    self.pila_frames = [main]

    # Pila de prep frames: ERA empuja, GOSUB activa
    self.pila_prep = []

    # IP arranca en 0
    self.IP = 0

"¿Por qué main es pila_frames[0] en vez de un caso especial?"

Regla 1 de las 3 reglas. Si tratamos main como caso especial ("aquí no hay frame"), el routing de leer/escribir tendría que ramificar: "si estoy en main usa memoria global, si estoy en una función usa frame actual". Con main como frame inicial, el routing queda uniforme: las direcciones 5000-11999 siempre van a pila_frames[-1].

4.6 Routing: leer / escribir / escribir_en_prep

leer(direccion)

def leer(self, direccion):
    seg_tipo = _segmento_y_tipo(direccion)
    if seg_tipo is None:
        raise RuntimeError(f"VM: direccion invalida al leer: {direccion}")
    seg, tipo = seg_tipo
    if seg == "cte":
        return self.memoria_constantes[direccion]
    if seg in ("global", "retorno"):           # <-- retorno alias de global
        return self.memoria_global.get(direccion, _DEFAULTS[tipo])
    f = self._frame_actual()
    store = f.locales if seg == "local" else f.temps
    return store.get(direccion, _DEFAULTS[tipo])

escribir(direccion, valor)

def escribir(self, direccion, valor):
    seg_tipo = _segmento_y_tipo(direccion)
    if seg_tipo is None:
        raise RuntimeError(f"VM: direccion invalida al escribir: {direccion}")
    seg, tipo = seg_tipo
    if seg == "cte":
        raise RuntimeError(f"VM: intento de escribir a una constante: {direccion}")
    if seg in ("global", "retorno"):           # <-- retorno alias de global
        self.memoria_global[direccion] = valor
        return
    f = self._frame_actual()
    store = f.locales if seg == "local" else f.temps
    store[direccion] = valor

escribir_en_prep(direccion, valor) — especial para PARAM

def escribir_en_prep(self, direccion, valor):
    """Escribe en el TOP de pila_prep (callee aun no activo)."""
    if not self.pila_prep:
        raise RuntimeError("VM: PARAM sin ERA previo")
    seg_tipo = _segmento_y_tipo(direccion)
    seg, _tipo = seg_tipo
    if seg != "local":
        raise RuntimeError(f"VM: PARAM destino no es local: {direccion}")
    self.pila_prep[-1].locales[direccion] = valor

"¿Por qué PARAM no usa escribir() normal?"

Regla 2 de las 3 reglas. Cuando se ejecuta PARAM, el "frame actual" es el CALLER (la función que invoca). El callee tiene su frame en pila_prep[-1] pero aún no está activo (eso pasa con GOSUB).

Si PARAM usara escribir() normal, los parametros del callee se escribirían en el frame del CALLER — sobreescribiendo las locales del caller cuando comparten números (que pasa siempre, porque cada función parte locales en 5000).

4.7 ejecutar() — loop principal

def ejecutar(self):
    LIMITE_FRAMES = 1000
    while self.IP < len(self.fila):
        q = self.fila[self.IP]
        op = q.op

        if op == "+":
            self.escribir(q.resultado, self.leer(q.opIzq) + self.leer(q.opDer))
        elif op == "-":
            self.escribir(q.resultado, self.leer(q.opIzq) - self.leer(q.opDer))
        # ...
        elif op == "GOTO":
            self.IP = q.resultado
            continue                              # <-- continue, no IP += 1
        elif op == "GOTOF":
            if not self.leer(q.opIzq):
                self.IP = q.resultado
                continue
        elif op == "ERA":
            self.pila_prep.append(self._nuevo_frame(q.resultado))
        elif op == "PARAM":
            self.escribir_en_prep(q.resultado, self.leer(q.opIzq))
        elif op == "GOSUB":
            prep = self.pila_prep.pop()
            prep.ip_retorno = self.IP + 1
            self.pila_frames.append(prep)
            if len(self.pila_frames) > LIMITE_FRAMES:
                raise RuntimeError(f"VM: stack overflow (>{LIMITE_FRAMES} frames)")
            self.IP = q.resultado
            continue
        elif op == "ENDFUNC":
            frame = self.pila_frames.pop()
            self.IP = frame.ip_retorno
            continue
        # ...
        self.IP += 1                              # <-- todos los no-saltos llegan aquí

Convención del IP

Una sola regla rige el avance del IP:

  • Opcodes que saltan (GOTO/GOTOF/GOSUB/ENDFUNC): setean IP y hacen continue.
  • Todos los demás: caen al IP += 1 al final del while.

Así ningún opcode necesita pensar "¿avanzo IP o no?".

4.9 Tips de examen

"¿Cómo soporta la VM las llamadas anidadas como f(g(x))?"

Regla 3: pila_prep es PILA. Al evaluar f(g(x)):

  1. ERA f -> push frame_f a pila_prep
  2. (evaluar g(x))
    1. ERA g -> push frame_g a pila_prep (ahora tiene [frame_f, frame_g])
    2. PARAM x -> escribe en TOP (frame_g)
    3. GOSUB g -> pop frame_g, push como activo
  3. PARAM g_result -> escribe en TOP (frame_f, que sigue ahí)
  4. GOSUB f -> pop frame_f

Sin pila, el ERA de g sobrescribiría el frame de f.

"¿Qué pasa con los defaults de variables no inicializadas?"

leer() hace store.get(direccion, _DEFAULTS[tipo]). Si la variable no fue escrita, devuelve _DEFAULTS["entero"]=0, _DEFAULTS["flotante"]=0.0, _DEFAULTS["bool"]=0. Esto evita crashes por KeyError y permite el "phantom temporal" (variables de retorno que no se escribieron).

"¿Por qué el límite de 1000 frames?"

Protección anti-runaway: si hay recursión infinita por caso base mal definido, detecta el problema y reporta error en vez de colgar el proceso o consumir toda la RAM. Cualquier programa Patito razonable vive cómodamente debajo del límite (factorial hace ~5 frames).

El bool en Patito es 0/1 (entero), no Python True/False. La VM hace int(...) al evaluar relacionales: self.escribir(q.resultado, int(leer(a) < leer(b))). Esto evita que True/False se filtre al output cuando haces escribe(a < b).
