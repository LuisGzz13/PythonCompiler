Auditoría visual de Puntos Neurálgicos

¿Cada uno es necesario y eficiente? · Patito Etapa 3 · Luis Manuel Gonzalez Martinez

Propósito de esta auditoría

Cada punto neurálgico (PN) del analizador semántico se examina aquí contra dos preguntas: ¿es necesario? (¿se podría omitir sin perder funcionalidad?) y ¿es eficiente? (¿hace solo el trabajo mínimo requerido?).

Para cada PN se muestra: un diagrama SVG del estado de las pilas antes y después de su ejecución, el código del override, qué pasaría si se eliminara, y un veredicto.

Necesario y eficiente
Parte obligada del algoritmo clásico de 3 pilas (Aho/Ullman); su trabajo es mantener invariantes de las pilas que el exit luego consume

Etapa 2: registro de scopes, variables y funciones

PN2-1 · enterPrograma

Necesario
Dispara: al entrar a la regla raíz programa, una sola vez por programa.
El directorio pasa de vacío a tener una entrada con el nombre del programa.
void enterPrograma(ProgramaContext* ctx) {
    auto* idNode = ctx->ID();
    nombrePrograma = idNode->getText();
    auto* entry = funcs.agregar(nombrePrograma, Tipo::NULA, linea);
    scopeActual = entry;
}

¿Qué pasaría sin él?

scopeActual sería nullptr al procesar las primeras declaraciones globales. enterDeclaracion haría early-return y NO se registraría ninguna variable global. El programa quedaría sin scope donde habitar.

¿Es eficiente?

Sí. Es la única forma de inicializar el scope global. No hay alternativa más barata.

PN2-2 · enterFuncs

Necesario
Dispara: al entrar a cada declaración de función. Una vez por función.
Se registra la nueva función + se mueve el scope activo. Etapa 3: además resetea contadores locales y temporales.
void enterFuncs(FuncsContext* ctx) {
    // ... validar duplicado ...
    scopeActual = nuevo;
    mem.resetScopeLocal();   // Etapa 3
    // Procesar parametros (los registra como vars locales con direccion).
}

¿Qué pasaría sin él?

Las funciones nunca se registrarían en el directorio. Sus parámetros y vars locales caerían en el scope global, generando duplicados y vars locales "globales". Etapa 3: las direcciones locales nunca se resetearían entre funciones, agotando rápido el rango.

¿Es eficiente?

Sí. Hace 3 cosas (registrar, switch scope, reset memoria), todas necesarias y ninguna duplicada.

PN2-3 · exitFuncs

Necesario
Dispara: al salir de una declaración de función. Una vez por función.
Restaura scope global para que las siguientes funciones/declaraciones se manejen correctamente.
void exitFuncs(FuncsContext* /*ctx*/) {
    scopeActual = funcs.obtener(nombrePrograma);
    enFuncDuplicada = false;
}

¿Qué pasaría sin él?

Las funciones DECLARADAS DESPUÉS quedarían registradas dentro del scope de la función anterior. Por ejemplo, definir nula f1(){} nula f2(){} haría que f2 y sus parámetros vivieran "dentro" de f1. Caos.

¿Es eficiente?

Sí. Dos asignaciones simples. No hay forma más barata de hacerlo.

PN2-4 · enterDeclaracion

Necesario
Dispara: al entrar a cada renglón id : tipo; dentro de un bloque vars.
Las dos vars se agregan a la tabla. Etapa 3: con su dirección asignada por AsignadorMemoria.
void enterDeclaracion(DeclaracionContext* ctx) {
    Tipo tipo = extraerTipo(ctx->tipo());
    bool esGlobal = (scopeActual->nombre == nombrePrograma);

    for (cada id en lista) {
        int direccion = esGlobal ? mem.nuevaGlobal(tipo) : mem.nuevaLocal(tipo);
        scopeActual->variables.agregar(nombre, tipo, linea, direccion);
        // valida duplicados via el return false de agregar
    }
}

¿Qué pasaría sin él?

Las variables nunca se registrarían. Toda referencia posterior a una variable fallaría con "variable no declarada". El programa entero se convertiría en un montón de errores.

¿Es eficiente?

Sí. Una iteración sobre la lista, una inserción por var. La asignación de dirección es O(1).

Etapa 3: generación de cuádruplos

PN3-A · exitFactor

Necesario
Dispara: al salir de cada <FACTOR>. Es el único lugar donde se empujan operandos atómicos a las pilas.
Cada operando atómico (variable, constante, llamada, expr en paréntesis) empuja su dirección y tipo a las pilas.
void exitFactor(FactorContext* ctx) {
    if (ctx->PIZQ()) return;             // ( expr ): ya empujado
    if (ctx->llamada()) { ... push phantom ... }
    if (ctx->ID())  { busca var; push direccion+tipo }
    if (ctx->cte()) { registra const; push direccion+tipo }
}

¿Qué pasaría sin él?

Las pilas siempre estarían vacías cuando PN3-B2/C2/D2 intenten popear operandos. Ningún cuádruplo se generaría. El programa "compila" sin código intermedio.

¿Es eficiente?

Sí. Cada factor produce exactamente un push (o nada en el caso de paréntesis donde el push ya ocurrió internamente).

PN3-B1 · enterExp (con MAS o MENOS)

Necesario por la rúbrica
Dispara: al entrar a una regla exp : exp '+' termino o exp : exp '-' termino.
Apila el operador para que PN3-B2 lo pueda popear al final. La pila acumula operadores anidados durante recursiones.
void enterExp(ExpContext* ctx) {
    if (ctx->MAS())   gen.pushOperador(Operador::MAS);
    if (ctx->MENOS()) gen.pushOperador(Operador::MENOS);
}

¿Qué pasaría sin él?

Sin este PN la pilaOperadores nunca recibiría +/-, por lo que PN3-B2 no podría hacer pop del operador esperado por el algoritmo clásico de 3 pilas. La invariante "el operador binario actual vive en el tope de la pila durante el procesamiento de sus operandos" se rompería.

¿Es eficiente?

Sí: una sola operación O(1) sobre la pila. Es el punto exacto donde el algoritmo de Aho/Ullman dicta empujar el operador (antes de evaluar los operandos) para preservar precedencia y asociatividad. Mantenerlo es lo que permite reescribir esta etapa hacia un parser tabular en el futuro sin tocar nada más del generador.

PN3-B2 · exitExp (con MAS o MENOS)

Necesario
Dispara: al salir de una exp que tenía operador, después de procesar ambos hijos.
Pop del operador y de los 2 operandos, validación con cubo, emisión del cuádruplo, push del resultado temporal.
void exitExp(ExpContext* ctx) {
    if (ctx->MAS() == nullptr && ctx->MENOS() == nullptr) return;
    Tipo res = gen.popYEmitirBinario();
    if (res == Tipo::ERROR) error("tipos incompatibles para '+/-'");
}

¿Qué pasaría sin él?

Las pilas crecerían sin generar cuádruplos. El programa "compila" sin código intermedio. Las pilas terminarían desbalanceadas al final del programa.

¿Es eficiente?

Sí. Es la única manera de emitir cuádruplos para suma/resta. Delega toda la complejidad a popYEmitirBinario, que es O(1).

PN3-C1 · enterTermino (con POR o ENTRE)

Algoritmo de 3 pilas
Dispara: al entrar a una regla termino : termino '*' factor o termino : termino '/' factor.
void enterTermino(TerminoContext* ctx) {
    if (ctx->POR())   gen.pushOperador(Operador::POR);
    if (ctx->ENTRE()) gen.pushOperador(Operador::ENTRE);
}

¿Qué pasaría sin él?

La pilaOperadores nunca recibiría *//, por lo que PN3-C2 no encontraría operador al hacer pop. La precedencia de *// sobre +/- deja de quedar asentada en la pila para casos anidados como a + b * c.

¿Es eficiente?

Sí: push en O(1). Es el dual exacto de PN3-B1 una nivel más profundo en la jerarquía de precedencia, y mantiene la simetría que pide el algoritmo clásico.

PN3-C2 · exitTermino (con POR o ENTRE)

Necesario
Dispara: al salir de una termino que tenía * o /.
void exitTermino(TerminoContext* ctx) {
    if (ctx->POR() == nullptr && ctx->ENTRE() == nullptr) return;
    Tipo res = gen.popYEmitirBinario();
    if (res == Tipo::ERROR) error("tipos incompatibles para '*' o '/'");
}

¿Qué pasaría sin él?

Multiplicaciones y divisiones nunca emitirían cuádruplos. a * b dejaría a y b en la pila sin procesar.

¿Es eficiente?

Sí. Mismo razonamiento que PN3-B2. Delega a popYEmitirBinario.

PN3-D1 · enterRelOpc (con opRel no-épsilon)

Algoritmo de 3 pilas
Dispara: al entrar a una relOpc que tiene un operador relacional (no épsilon).
void enterRelOpc(RelOpcContext* ctx) {
    auto* opRel = ctx->opRel();
    if (opRel == nullptr) return;
    if (opRel->MAYOR())    gen.pushOperador(Operador::MAYOR);
    if (opRel->MENOR())    gen.pushOperador(Operador::MENOR);
    if (opRel->DISTINTO()) gen.pushOperador(Operador::DISTINTO);
    if (opRel->IGUAL2())   gen.pushOperador(Operador::IGUAL2);
}

¿Qué pasaría sin él?

Sin este PN, PN3-D2 no tendría operador relacional en la pila al cierre de la expresión. El cuádruplo relacional no podría emitirse usando la misma rutina popYEmitirBinario que usan los aritméticos: habría que duplicar la lógica leyendo el operador desde el contexto sintáctico.

¿Es eficiente?

Sí: un push O(1). Mantiene la simetría con PN3-B1 y PN3-C1, lo que permite reutilizar popYEmitirBinario sin ramificar por familia de operador.

PN3-D2 · exitExpresion (con relOpc no-épsilon)

Necesario
Dispara: al terminar una expresion que tenía relOpc. Es donde se emite el cuádruplo relacional.
void exitExpresion(ExpresionContext* ctx) {
    auto* relOpc = ctx->relOpc();
    if (relOpc == nullptr || relOpc->opRel() == nullptr) return;
    Tipo res = gen.popYEmitirBinario();
    if (res == Tipo::ERROR) error("tipos incompatibles en expresion relacional");
}

¿Qué pasaría sin él?

Comparaciones a < b nunca emitirían cuádruplos. Los si y mientras (Etapa 4) no tendrían el BOOL para decidir saltos.

¿Es eficiente?

Sí. Single check del context, delega a popYEmitirBinario, valida cubo.

PN3-E · exitAsigna

Necesario
Dispara: al terminar de procesar id = expresion ;. El RHS ya dejó su resultado en pilaOperandos.
void exitAsigna(AsignaContext* ctx) {
    // Resuelve destino, pop RHS, valida cubo, emite '='
    const VariableInfo* v = buscarVar(nombre);
    if (v == nullptr) { error("no declarada"); ... return; }
    Tipo res = gen.emitirAsignacion(v->direccion, v->tipo);
    if (res == Tipo::ERROR) error("tipos incompatibles en asignacion");
}

¿Qué pasaría sin él?

El RHS quedaría en la pila sin asignarse a la variable destino. No habría cuádruplos de asignación. Las vars nunca cambiarían en runtime.

¿Es eficiente?

Sí. Una sola búsqueda de la variable destino y un emit. Sin redundancia.

PN3-F · exitImpElem

Necesario
Dispara: al salir de cada elemento de la lista del escribe(...).
void exitImpElem(ImpElemContext* ctx) {
    if (ctx->LETRERO()) {
        int dir = cte.direccionDeLetrero(lex);
        gen.emitirPrint(dir);
    } else {  // era expresion
        int dir = gen.topOperando();
        gen.emitirPrint(dir);
        gen.popOperando();
    }
}

¿Qué pasaría sin él?

Ningún cuádruplo PRINT se emitiría. Los letreros se registrarían (si otro PN los manejara) pero no se imprimirían en ejecución.

¿Es eficiente?

Sí. Un solo cuádruplo por elemento. La distinción letrero vs expresión es necesaria porque vienen de distinto lado de la gramática.

Veredicto global

De los 13 puntos neurálgicos implementados:

  • 10 son estrictamente necesarios (badge verde): sin ellos el compilador deja de funcionar.
  • 3 implementan la mecánica clásica de la pilaOperadores (badge amarillo): PN3-B1, PN3-C1, PN3-D1 — los enter que empujan el operador binario antes de procesar sus operandos. Son los puntos exactos donde el algoritmo de Aho/Ullman dicta hacer el push; sin ellos PN3-B2, PN3-C2 y PN3-D2 no podrían usar popYEmitirBinario de forma uniforme y la invariante de la pila de operadores se perdería.

Por qué la auditoría los marca aparte (badge amarillo):

  1. Son puntos de preparación: no emiten cuádruplo por sí mismos, solo dejan la pila lista para que el exit correspondiente lo haga. Esto los hace conceptualmente distintos de los exit verdes, que sí ejecutan el trabajo observable.
  2. El badge amarillo destaca que la rúbrica pide explícitamente las 3 pilas (operandos, tipos, operadores) — esta familia de PNs es la que materializa esa tercera pila.
  3. Mantener el patrón "push en enter, pop en exit" hace el código simétrico y trivial de extender. Cuando Etapa 4 agregue saltos a parchar para si/mientras, los nuevos PNs encajan en la misma forma.

Conclusión: el conjunto de puntos neurálgicos es completo y defensible. Cada PN tiene un propósito documentado, una invariante que mantiene sobre las pilas, y un consumidor claro aguas abajo. Ninguno se eliminó tras la auditoría.
