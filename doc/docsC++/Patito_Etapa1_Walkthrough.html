Etapa 1 · Scanner + Parser con ANTLR 4 (runtime C++) · Luis Manuel Gonzalez Martinez · A01722501

Visión general

El proyecto recibe un archivo con código en el lenguaje Patito (extensión .patito) y verifica que esté léxica y sintácticamente bien formado. Para lograrlo se usa la siguiente cadena:

Pipeline:
archivo.patito → PatitoLexer (tokens) → PatitoParser (árbol de parseo) → main.cpp reporta OK / FAIL

Hay cuatro archivos hechos a mano:

  • grammar/Patito.g4 — la gramática del lenguaje en formato ANTLR4. Contiene tanto las reglas léxicas (qué es un token) como las sintácticas (cómo se combinan). ANTLR lee este archivo y genera el lexer y el parser en C++.
  • src/main.cpp — el driver que lee el archivo de entrada, llama al lexer y al parser, y reporta resultados con línea y columna.
  • CMakeLists.txt — la receta de compilación que indica a CMake qué archivos compilar, dónde está el runtime de ANTLR, y cómo enlazar todo.
  • tests/run_tests.sh — el runner del test-plan que ejecuta los 24 programas de prueba y verifica que cada uno pase o falle como se espera.

El resto de los archivos en src/generated/ los produce ANTLR automáticamente — no los tocamos.

grammar/Patito.g4

grammar/Patito.g4
Gramática unificada (lexer + parser) en formato ANTLR4. Es la traducción directa de la CFG y las expresiones regulares de Etapa 0.

ANTLR distingue las dos clases de reglas por la primera letra del nombre:

  • Reglas de parser (no-terminales) → empiezan con minúscula. Describen cómo se combinan los tokens para formar construcciones del lenguaje (programa, declaración, expresión...).
  • Reglas de lexer (terminales/tokens) → empiezan con MAYÚSCULA. Describen los caracteres que forman cada token (palabras reservadas, números, identificadores...).

Cabecera

1grammar Patito;

Declara que esta gramática se llama Patito. ANTLR exige que el nombre coincida con el del archivo (Patito.g4). Este nombre también se convierte en el prefijo de las clases generadas: PatitoLexer, PatitoParser, PatitoVisitor.

Como no especificamos parser grammar ni lexer grammar, ANTLR la trata como combinada: en este mismo archivo conviven las reglas léxicas y las sintácticas.

3// ================================================================= 4// PARSER RULES (no-terminales) 5// Traduccion 1:1 de la CFG de Etapa 0 6// =================================================================

Comentarios. Indican el inicio de la sección de reglas de parser. Los comentarios // ... se permiten en cualquier parte de un .g4 y ANTLR los ignora. Aquí los uso para que el archivo sea legible: separa visualmente la zona del parser de la del lexer (que viene más abajo en la línea 188).

programa (la regla raíz)

8programa 9 : PROGRAMA ID PCOMA varsOpc funcsOpc INICIO cuerpo FIN 10 ;

Esta es la regla raíz: el parser siempre empieza por aquí. Define la forma de un programa Patito completo, traduciendo el diagrama <Programa> de la rúbrica.

La línea 9 dice: para que un texto sea un programa válido, debe ser exactamente la secuencia:

  • PROGRAMA — la palabra reservada programa
  • ID — el nombre del programa
  • PCOMA — el ;
  • varsOpc — opcionalmente un bloque vars
  • funcsOpc — opcionalmente cero o más funciones
  • INICIO — la palabra reservada inicio
  • cuerpo — el bloque { ... } del programa principal
  • FIN — la palabra reservada fin

El : introduce la(s) producción(es) y el ; de la línea 10 cierra la regla. Esa sintaxis se repite en todas las reglas del archivo.

12varsOpc 13 : vars 14 | 15 ;

Define una versión opcional del bloque vars. Tiene dos alternativas separadas por |:

  • Línea 13: vars — usar el bloque vars completo
  • Línea 14: nada (después del | no hay tokens) — eso es épsilon, la cadena vacía

Por eso un programa puede tener o no tener declaraciones globales. La rúbrica lo permite y el diagrama lo refleja con la flecha que se salta el bloque VARS.

17funcsOpc 18 : funcs funcsOpc 19 | 20 ;

Permite cero o más funciones antes de inicio. Es una regla recursiva por la derecha:

  • Línea 18: una función seguida de "más cosas que también son funciones opcionales".
  • Línea 19: épsilon, que corta la recursión.

Si el programa no tiene funciones, esta regla se reduce inmediatamente a épsilon. Si tiene tres funciones, se expande tres veces antes de cortar.

Vars y declaraciones

22vars 23 : VARS declaracionList 24 ;

Bloque vars: la palabra reservada seguida de una lista de declaraciones. La regla no admite alternativa épsilon porque si llegamos aquí ya decidimos que SÍ hay vars (la opción de "no vars" la maneja varsOpc).

26declaracionList 27 : declaracion declaracionList 28 | declaracion 29 ;

Una o más declaraciones (por lo menos una, porque el caso "lista vacía" no tiene sentido tras la palabra vars). Patrón estándar:

  • Línea 27: una declaración seguida de la lista (recursión por la derecha)
  • Línea 28: solo una declaración (caso base)
31declaracion 32 : idLista DPUNTOS tipo PCOMA 33 ;

Una declaración tiene la forma a, b, c : entero ;:

  • idLista — uno o más identificadores separados por coma
  • DPUNTOS — el :
  • tipo — entero o flotante
  • PCOMA — el ; que cierra la declaración
35idLista 36 : ID idResto 37 ; 38 39idResto 40 : COMA ID idResto 41 | 42 ;

Esta es la traducción del bucle id (, id)* que aparece en el diagrama de <VARS>. La idea es:

  • idLista = exige al menos un ID, seguido del "resto"
  • idResto = cero o más ", ID" después del primero. La recursión por la derecha permite encadenarlos: , b , c , d.

Esta forma con dos reglas auxiliares (idLista/idResto) corresponde a una traducción "manual" del bucle del diagrama, similar a como se haría en una gramática LL(1) clásica. ANTLR4 también permite escribir ID (',' ID)* con menos código, pero mantengo este estilo para que el .g4 mapee 1:1 con la CFG documentada en Etapa 0.

44tipo 45 : ENTERO 46 | FLOTANTE 47 ;

Los únicos dos tipos del lenguaje. Cualquier intento de declarar otro tipo (por ejemplo cadena) cae en error porque el lexer leería cadena como un ID y el parser exige aquí ENTERO o FLOTANTE (token de palabra reservada). El test 05_tipo_invalido.patito prueba precisamente eso.

Funciones y llamadas

49funcs 50 : tipoFunc ID PIZQ paramsOpc PDER LLAVEIZQ varsOpc cuerpo LLAVEDER PCOMA 51 ;

Forma completa de una declaración de función:

  • tipoFunc — el tipo de retorno (nula, entero o flotante)
  • ID — nombre de la función
  • PIZQ ... PDER — paréntesis que envuelven los parámetros
  • paramsOpc — opcionalmente la lista de parámetros
  • LLAVEIZQ ... LLAVEDER — llaves que envuelven todo el cuerpo de la función
  • varsOpc — vars locales (opcionales)
  • cuerpo — el bloque { estatutos } con el código
  • PCOMA — el ; final que cierra la declaración (lo exige el diagrama)

Nota la doble llave: la función se define como { varsOpc cuerpo } donde cuerpo a su vez es { estatutos }. Por eso en los tests las funciones se ven como nombre() { { ... } } ;. Esto viene directo del diagrama de <FUNCS> — las llaves externas envuelven vars+cuerpo, las internas son las del cuerpo.

53tipoFunc 54 : NULA 55 | tipo 56 ;

El retorno puede ser nula (sin valor) o uno de los tipos básicos. Reusa la regla tipo en lugar de listar ENTERO | FLOTANTE, para evitar duplicación.

58paramsOpc 59 : paramLista 60 | 61 ; 62 63paramLista 64 : ID DPUNTOS tipo paramResto 65 ; 66 67paramResto 68 : COMA ID DPUNTOS tipo paramResto 69 | 70 ;

Misma estrategia que idLista/idResto, pero cada parámetro lleva además : tipo. Soporta:

  • Cero parámetros: paramsOpc elige épsilon → foo()
  • Uno: paramLista con paramResto en épsilon → foo(x : entero)
  • Varios: el paramResto se repite → foo(a : entero, b : flotante, c : entero)
72llamada 73 : ID PIZQ argsOpc PDER 74 ;

Una llamada a función: nombre, paréntesis y lista opcional de argumentos. Importante: sin punto y coma, porque la llamada también puede aparecer dentro de una expresión (ver factor). Cuando se usa como estatuto, el ; lo agrega estatuto en la línea 103 (llamada PCOMA).

76argsOpc 77 : argLista 78 | 79 ; 80 81argLista 82 : expresion argResto 83 ; 84 85argResto 86 : COMA expresion argResto 87 | 88 ;

Lista de argumentos en una llamada. Cada argumento es una expresion completa, no solo un identificador o constante. Por eso uno puede pasar imprimir(a + 1, b * 2): cada argumento se parsea con todo el árbol de expresiones, operadores y paréntesis.

Cuerpo y estatutos

90cuerpo 91 : LLAVEIZQ estatutoList LLAVEDER 92 ; 93 94estatutoList 95 : estatuto estatutoList 96 | 97 ;

Un cuerpo es un par de llaves con cero o más estatutos dentro. estatutoList permite épsilon, así que un cuerpo vacío { } es válido (lo prueba 01_minimo.patito).

99estatuto 100 : asigna 101 | condicion 102 | ciclo 103 | llamada PCOMA 104 | imprime 105 ;

Las cinco formas que puede tomar un estatuto, exactamente como aparecen en el diagrama de <ESTATUTO>:

  • asigna — x = expr ;
  • condicion — si (...) { } sino { } ;
  • ciclo — mientras (...) haz { } ;
  • llamada PCOMA — una llamada a función como sentencia, con el ; agregado aquí (no es parte de llamada porque también se usa dentro de expresiones)
  • imprime — escribe(...) ;

Asignación, condición y ciclo

107asigna 108 : ID IGUAL expresion PCOMA 109 ;

Asignación clásica: id = expresion ;. La parte derecha es una expresion (no un simple literal), lo cual permite cosas como x = a + b * c;.

111condicion 112 : SI PIZQ expresion PDER cuerpo sinoOpc PCOMA 113 ; 114 115sinoOpc 116 : SINO cuerpo 117 | 118 ;

Condicional si:

  • SI PIZQ expresion PDER — el si (...)
  • cuerpo — el bloque que se ejecuta si la condición es verdadera
  • sinoOpc — opcionalmente un sino { ... }
  • PCOMA — el ; final, que el diagrama exige al cerrar el si

Por esto los tests muestran si (...) { ... } ; con el punto y coma al final. Si lo omites cae en error: lo prueba 10_si_sin_pcoma.patito.

120ciclo 121 : MIENTRAS PIZQ expresion PDER HAZ cuerpo PCOMA 122 ;

Bucle mientras (...) haz { ... } ;. La palabra reservada haz conecta la condición con el cuerpo, tal como aparece en el diagrama. El ; al final cierra el ciclo, igual que en si.

Imprime

124imprime 125 : ESCRIBE PIZQ impLista PDER PCOMA 126 ; 127 128impLista 129 : impElem impResto 130 ; 131 132impResto 133 : COMA impElem impResto 134 | 135 ; 136 137impElem 138 : expresion 139 | LETRERO 140 ;

El escribe imprime uno o más elementos separados por coma. Cada elemento (impElem) puede ser:

  • Una expresion aritmética/relacional (que se evaluará).
  • Un LETRERO (cadena literal entre comillas dobles), que se imprime tal cual.

Por eso son válidas tanto escribe("Hola"); como escribe("x vale ", x + 1, " unidades");. Esta mezcla viene del diagrama de <IMPRIME>.

Expresiones

142expresion 143 : exp relOpc 144 ; 145 146relOpc 147 : opRel exp 148 | 149 ; 150 151opRel 152 : MAYOR 153 | MENOR 154 | DISTINTO 155 | IGUAL2 156 ;

Una expresion es una exp aritmética seguida opcionalmente de un operador relacional y otra exp. Así modelo:

  • Una expresión simple: a + b (relOpc en épsilon).
  • Una comparación: a + b < c * 2 (un solo nivel de comparación, sin encadenar).

Esto refleja directo el diagrama de <EXPRESIÓN> y limita a una sola comparación por expresión, lo cual es típico en lenguajes simples (no se permite a < b < c). Los cuatro operadores relacionales son >, <, != y ==.

158exp 159 : exp MAS termino 160 | exp MENOS termino 161 | termino 162 ; 163 164termino 165 : termino POR factor 166 | termino ENTRE factor 167 | factor 168 ;

Suma/resta y multiplicación/división. Estas son recursivas por la izquierda: el lado izquierdo de la regla aparece como primer símbolo de la producción. Eso le da al parser asociatividad izquierda:

Por ejemplo, 10 - 3 - 2 se interpreta como (10 - 3) - 2 = 5, no como 10 - (3 - 2) = 9.

Y la precedencia aparece naturalmente porque termino está "más adentro" que exp: en 1 + 2 * 3, el 2 * 3 es un termino y se reduce primero, dando 1 + 6 = 7.

ANTLR4 acepta esta forma directa, internamente la reescribe sin perder asociatividad. En contraste, un parser LL(1) puro requeriría reescribirlo a exp → termino exp', etc. Ese es uno de los motivos para escoger ANTLR4.

170factor 171 : PIZQ expresion PDER 172 | llamada 173 | signoOpc ID 174 | signoOpc cte 175 ;

Las cuatro formas posibles de un operando "atómico" en una expresión:

  • PIZQ expresion PDER — una expresión entre paréntesis, que vuelve a entrar al ciclo recursivo.
  • llamada — una llamada a función como operando, p. ej. obtener() + 1.
  • signoOpc ID — un identificador, opcionalmente precedido por + o -: x, -x, +x.
  • signoOpc cte — una constante (entera o flotante) con signo unario opcional: 5, -5, 3.14.

Por qué este orden: tanto llamada como signoOpc ID empiezan con ID (en el caso de signoOpc en épsilon). Para evitar ambigüedad, listo llamada antes de signoOpc ID: cuando ANTLR ve nombre ( entiende que es una llamada; cuando ve nombre sin paréntesis adelante cae en signoOpc ID. ANTLR4 LL(*) maneja esta predicción automáticamente con varios tokens de lookahead.

177signoOpc 178 : MAS 179 | MENOS 180 | 181 ; 182 183cte 184 : CTE_ENT 185 | CTE_FLOT 186 ;

signoOpc permite un signo unario + o - opcional (la tercera alternativa épsilon equivale a "sin signo"). Solo aplica delante de ID o de cte, no delante de paréntesis ni de llamadas, tal como aparece en el diagrama de <FACTOR>.

cte reúne los dos tipos de constantes numéricas: entera y flotante. Definir esto como una regla aparte permite que factor trate ambos casos con una sola alternativa (signoOpc cte).

Lexer: palabras reservadas

188// ================================================================= 189// LEXER RULES (tokens) 190// Las palabras reservadas van ANTES de ID para que ganen el match 191// ================================================================= 192 193// Palabras reservadas 194PROGRAMA : 'programa' ; 195INICIO : 'inicio' ; 196FIN : 'fin' ; 197VARS : 'vars' ; 198ENTERO : 'entero' ; 199FLOTANTE : 'flotante' ; 200ESCRIBE : 'escribe' ; 201SI : 'si' ; 202SINO : 'sino' ; 203MIENTRAS : 'mientras' ; 204HAZ : 'haz' ; 205NULA : 'nula' ;

Cada línea declara un token de palabra reservada. La sintaxis es NOMBRE : 'literal' ;. Cuando el lexer encuentra exactamente ese literal en el código fuente, emite un token con ese nombre.

Por qué van antes que ID (línea 229): ANTLR resuelve los empates entre reglas de lexer por orden de aparición. La palabra programa también encajaría con ID (es una secuencia de letras), pero como PROGRAMA está declarado primero, gana. Si invirtiera el orden, programa, inicio, etc., serían reconocidos como identificadores y todo el parser fallaría. El test 02_keyword_como_id.patito verifica que efectivamente no se pueda usar entero como identificador.

Lexer: operadores

207// Operadores 208MAS : '+' ; 209MENOS : '-' ; 210POR : '*' ; 211ENTRE : '/' ; 212MAYOR : '>' ; 213MENOR : '<' ; 214DISTINTO : '!=' ; 215IGUAL2 : '==' ; 216IGUAL : '=' ;

Los operadores aritméticos (+ - * /), relacionales (> < != ==) y el de asignación (=). Cada uno se asocia a un nombre simbólico que las reglas del parser consumen.

Punto sutil: IGUAL2 (==) tiene dos caracteres y IGUAL (=) uno solo. ANTLR siempre prefiere el match más largo que sea posible. Cuando ve == reconoce IGUAL2, no IGUAL IGUAL. Lo mismo para != que se reconoce como DISTINTO y no como otra cosa.

Lexer: puntuación

218// Puntuacion 219PCOMA : ';' ; 220COMA : ',' ; 221DPUNTOS : ':' ; 222PIZQ : '(' ; 223PDER : ')' ; 224LLAVEIZQ : '{' ; 225LLAVEDER : '}' ;

Los siete signos de puntuación que el lenguaje usa: punto y coma, coma, dos puntos, paréntesis y llaves. Cada uno en su propio token para que las reglas del parser los referencien por nombre simbólico.

Lexer: ID y literales

227// Identificadores y literales 228// CTE_FLOT debe ir antes que CTE_ENT para que 3.14 no se parsee como CTE_ENT '.' CTE_ENT 229ID : [a-zA-Z] [a-zA-Z0-9_]* ; 230CTE_FLOT : [0-9]+ '.' [0-9]+ ; 231CTE_ENT : [0-9]+ ; 232LETRERO : '"' ~["\r\n]* '"' ;

Las cuatro reglas léxicas más interesantes:

  • ID (línea 229): empieza con una letra (mayúscula o minúscula), seguida de cero o más letras, dígitos o guiones bajos. Por eso x, contador, i_2 son válidos pero _x o 2x no.
  • CTE_FLOT (línea 230): uno o más dígitos, un punto literal, uno o más dígitos. Exige al menos un dígito a cada lado del punto, así que 3.14 es válido pero .5 o 3. no.
  • CTE_ENT (línea 231): uno o más dígitos.
  • LETRERO (línea 232): una comilla doble, cero o más caracteres que no sean comilla, retorno de carro ni salto de línea, y otra comilla doble. La sintaxis ~["\r\n] es la negación de clase de ANTLR. Por eso un letrero no puede contener saltos de línea ni comillas internas (no hay escapes).

Por qué CTE_FLOT va antes que CTE_ENT: con la entrada 3.14, ambos podrían empezar a hacer match. Si CTE_ENT estuviera primero, ANTLR podría intentar reconocer 3 como entero, luego . como token desconocido, etc. Al poner CTE_FLOT primero, el lexer intenta la regla más específica antes y, gracias al match más largo, prefiere los 4 caracteres 3.14 sobre el solo 3.

Lexer: whitespace

234// Espacios en blanco: se descartan 235WS : [ \t\r\n]+ -> skip ;

Cualquier secuencia de espacio, tabulador, retorno de carro o salto de línea es un token WS ... pero -> skip le dice a ANTLR que lo descarte en vez de pasarlo al parser. Sin esto, el parser tendría que considerar espacios entre cada par de tokens, lo cual no tiene sentido aquí.

Esto explica también por qué los programas Patito pueden tener formato libre: indentación, líneas en blanco y separación entre tokens son irrelevantes mientras la secuencia de tokens (no whitespace) sea la correcta.

Notación: en las reglas léxicas, [a-z] es una clase de caracteres, + es "uno o más", * es "cero o más", ~[X] es la negación de la clase, 'literal' es un carácter literal. Para el punto literal usamos comillas ('.') porque . sin comillas significa "cualquier carácter".

Clase ColectorErrores

17class ColectorErrores : public BaseErrorListener { 18public: 19 int conteo = 0; 20 const std::string fuente; 21 explicit ColectorErrores(std::string fuente) : fuente(std::move(fuente)) {}

Defino una subclase de BaseErrorListener para capturar los errores de parsing y lexing con un formato propio en lugar del de ANTLR.

Función imprimirTokens

33static void imprimirTokens(PatitoLexer &lexer, CommonTokenStream &tokens) { 34 std::cout << "=== TOKENS ===" << std::endl; 35 auto &vocab = lexer.getVocabulary();

Función auxiliar que imprime la lista de tokens (la usa la bandera --tokens). static limita su visibilidad a este archivo (no se exporta).

Línea 35: obtiene el "vocabulario" del lexer. El vocabulario es la tabla que mapea cada tipo numérico de token (1, 2, 3, ...) a su nombre simbólico (PROGRAMA, ID, ...). Sin él, los tipos serían solo números.

  • Líneas 40–41: extraigo el lexema (texto fuente del token) y lo limpio si es <EOF> (porque ANTLR pone ese texto sintético).
42 std::cout << " linea " << tk->getLine() << ":" 43 << tk->getCharPositionInLine() << "\t" 44 << nombre << "\t'" << lex << "'" << std::endl; 45 } 46 std::cout << std::endl; 47}

Imprime una línea por token: línea, columna, nombre simbólico y lexema entre comillas. Por ejemplo:

  linea 1:0   PROGRAMA    'programa'
  linea 1:9   ID          'hola'

Esto demuestra que el scanner separa correctamente el código en tokens. Es la salida que se entregaría como evidencia del análisis léxico.

main: parseo de argumentos

49int main(int argc, char *argv[]) { 50 bool mostrarTokens = false; 51 bool mostrarArbol = false; 52 std::string archivo;

Función main con la firma estándar de C++. Las dos banderas (--tokens, --tree) y el nombre del archivo se inicializan en falso/vacío y se llenan con el bucle siguiente.

54 for (int i = 1; i < argc; i++) { 55 std::string arg = argv[i]; 56 if (arg == "--tokens") { 57 mostrarTokens = true; 58 } else if (arg == "--tree") { 59 mostrarArbol = true; 60 } else if (arg.rfind("--", 0) == 0) { 61 std::cerr << "Bandera desconocida: " << arg << std::endl; 62 return 2; 63 } else { 64 archivo = arg; 65 } 66 }

Recorre los argumentos a partir del índice 1 (el 0 es el nombre del programa).

  • Línea 56: si es --tokens, activa la bandera correspondiente.
  • Línea 58: lo mismo para --tree.
  • Línea 60: si empieza con -- pero no es ninguna conocida, error inmediato. arg.rfind("--", 0) == 0 es la forma idiomática de "starts with" en C++17.
  • Línea 64: cualquier argumento no-bandera se considera el nombre del archivo.

El return 2 en el caso de bandera desconocida sigue una convención común: 0 = OK, 1 = errores en el código, 2 = error de uso del binario.

main: apertura de archivo

68 if (archivo.empty()) { 69 std::cerr << "Uso: patito [--tokens] [--tree] archivo.patito" << std::endl; 70 return 2; 71 } 72 73 std::ifstream entrada(archivo); 74 if (!entrada.is_open()) { 75 std::cerr << "No se pudo abrir " << archivo << std::endl; 76 return 2; 77 }

Validaciones básicas:

  • Líneas 68–71: si no me pasaron archivo, imprimo el uso y salgo con error de uso.
  • Línea 73: abre el archivo en modo lectura (std::ifstream).
  • Líneas 74–77: si no se pudo abrir (no existe, sin permisos...), aviso y salgo.

main: pipeline lexer→parser

79 ANTLRInputStream input(entrada); 80 PatitoLexer lexer(&input);

Aquí empieza la cadena de ANTLR.

  • Línea 79: envuelve el std::ifstream en un ANTLRInputStream, que es la abstracción de "fuente de caracteres" que el lexer entiende.
  • Línea 80: instancia el PatitoLexer apuntando a esa fuente. Es la clase generada a partir de las reglas léxicas de Patito.g4.
82 ColectorErrores erroresLex(archivo); 83 lexer.removeErrorListeners(); 84 lexer.addErrorListener(&erroresLex);

Reemplazo el reporter de errores por defecto del lexer:

  • Línea 82: instancia mi colector con el nombre del archivo.
  • Línea 83: quita el listener por defecto (que imprime con un formato distinto al mío y a std::cerr directamente).
  • Línea 84: registra mi colector. A partir de aquí, cualquier error léxico (por ejemplo, una comilla sin cerrar) pasará por ColectorErrores::syntaxError.
86 CommonTokenStream tokens(&lexer); 87 tokens.fill();

Línea 86: crea un CommonTokenStream, que es el "buffer" de tokens entre el lexer y el parser.

Línea 87: fill() fuerza al lexer a leer todo el archivo y producir todos los tokens de una sola vez. Es necesario hacerlo aquí (no es estrictamente requerido por ANTLR) para que imprimirTokens tenga la lista completa antes de pasar al parser.

89 if (mostrarTokens) { 90 imprimirTokens(lexer, tokens); 91 }

Si la bandera --tokens estaba activa, llama a la función auxiliar definida arriba para volcar la lista de tokens.

93 PatitoParser parser(&tokens); 94 ColectorErrores erroresPar(archivo); 95 parser.removeErrorListeners(); 96 parser.addErrorListener(&erroresPar);

Misma idea que con el lexer pero ahora para el parser:

  • Línea 93: instancia el parser sobre el stream de tokens.
  • Líneas 94–96: cambia el listener de errores por uno propio. Uso dos colectores separados (erroresLex y erroresPar) para poder sumar sus conteos al final y saber si hubo errores en alguna fase.
98 auto *arbol = parser.programa();

Esta es la línea clave: arranca el parsing desde la regla programa (la regla raíz definida en Patito.g4). El parser consume tokens, intenta derivarlos según la gramática y devuelve un árbol de parseo.

Si todo va bien, arbol apunta a un árbol válido. Si hay errores, ANTLR igual devuelve un árbol parcial (con nodos de error), pero los listeners ya habrán anotado los problemas en erroresPar.

100 if (mostrarArbol) { 101 std::cout << "=== ARBOL DE PARSEO ===" << std::endl; 102 std::cout << arbol->toStringTree(&parser, true) << std::endl; 103 std::cout << std::endl; 104 }

Si la bandera --tree está activa, imprime el árbol de parseo en formato LISP. toStringTree(&parser, true) con true usa nombres legibles de las reglas (en lugar de números). El resultado se ve como (programa programa hola ; varsOpc funcsOpc inicio (cuerpo { estatutoList }) fin). Útil para verificar visualmente la estructura.

main: resultado y exit

106 int total = erroresLex.conteo + erroresPar.conteo; 107 if (total == 0) { 108 std::cout << "OK " << archivo << std::endl; 109 return 0; 110 } 111 std::cerr << "FAIL " << archivo << " (" << total << " error" 112 << (total == 1 ? "" : "es") << ")" << std::endl; 113 return 1; 114}

Suma los errores encontrados por ambos colectores. Si fueron cero, imprime OK y exit 0. Si hubo errores, imprime FAIL con el conteo (singular o plural según el caso) y exit 1.

Estos códigos de salida son los que el run_tests.sh verifica: 0 para programas válidos, 1 para programas inválidos. Si en el futuro la rúbrica pide otro convenio, se cambia aquí.

CMakeLists.txt

CMakeLists.txt
Receta de compilación. Le dice a CMake cómo localizar el runtime de ANTLR, qué archivos compilar y cómo enlazarlos en el binario patito.
1cmake_minimum_required(VERSION 3.20) 2project(Patito CXX)

Línea 1: exige CMake 3.20 o superior, que es donde ya están bien soportados los targets modernos. La versión instalada por Homebrew es 4.3.2, así que sobra.

Línea 2: declara un proyecto llamado Patito que usa CXX (C++) como único lenguaje. Sin esto, CMake intentaría detectar también un compilador de C, lo cual no necesitamos.

4set(CMAKE_CXX_STANDARD 17) 5set(CMAKE_CXX_STANDARD_REQUIRED ON)

Fija el estándar a C++17:

  • Línea 4: pide C++17.
  • Línea 5: REQUIRED ON hace que sea un error fatal si el compilador no lo soporta (en lugar de bajar silenciosamente al estándar anterior).

El runtime de ANTLR 4.13 requiere mínimo C++17, así que esto es lo más bajo que podemos pedir.

7find_path(ANTLR4_INCLUDE_DIR 8 NAMES antlr4-runtime.h 9 HINTS /opt/homebrew/include /usr/local/include 10 PATH_SUFFIXES antlr4-runtime 11)

Le pide a CMake que busque el directorio que contiene el header antlr4-runtime.h. Estrategia:

  • Línea 8: el archivo a localizar.
  • Línea 9: directorios donde mirar primero. /opt/homebrew/include es la ruta de Homebrew en Apple Silicon; /usr/local/include es la de Intel y de instalaciones manuales.
  • Línea 10: dentro de esas rutas, prueba el subdirectorio antlr4-runtime/, que es donde Homebrew deja los headers.

El resultado se guarda en la variable ANTLR4_INCLUDE_DIR.

13find_library(ANTLR4_LIB 14 NAMES antlr4-runtime 15 HINTS /opt/homebrew/lib /usr/local/lib 16)

Misma idea pero ahora para la librería compilada. Busca libantlr4-runtime.dylib (o .a) en los directorios típicos de Homebrew. Resultado en ANTLR4_LIB.

18if(NOT ANTLR4_INCLUDE_DIR OR NOT ANTLR4_LIB) 19 message(FATAL_ERROR "ANTLR4 C++ runtime no encontrado. Instala con: brew install antlr4-cpp-runtime") 20endif() 21 22message(STATUS "ANTLR4 include: ${ANTLR4_INCLUDE_DIR}") 23message(STATUS "ANTLR4 lib: ${ANTLR4_LIB}")

Líneas 18–20: si CMake no encontró el header o la librería, aborta con un mensaje claro que dice qué hace falta instalar. Esto evita errores crípticos de compilación más adelante (cosas como fatal error: 'antlr4-runtime.h' not found).

Líneas 22–23: imprime las rutas encontradas como información, útil al configurar para confirmar que tomó las rutas correctas.

25file(GLOB GENERATED_SRC ${CMAKE_SOURCE_DIR}/src/generated/*.cpp)

Recoge todos los archivos .cpp dentro de src/generated/ en una variable. Esos son los que ANTLR genera (PatitoLexer.cpp, PatitoParser.cpp, PatitoVisitor.cpp, etc.).

Con file(GLOB ...) no tengo que listarlos uno por uno — si en el futuro ANTLR genera un archivo nuevo, lo recogerá automáticamente. La advertencia clásica de CMake sobre GLOB es que no detecta cuando agregas archivos sin reconfigurar; aquí no es un problema porque cada vez que cambias Patito.g4 regeneras y reconfiguras de todas formas.

27add_executable(patito 28 src/main.cpp 29 ${GENERATED_SRC} 30)

Define el ejecutable patito. Sus fuentes son mi main.cpp (línea 28) y todos los .cpp generados por ANTLR (línea 29). El binario resultante se llama patito y se deposita en build/patito.

32target_include_directories(patito PRIVATE 33 ${ANTLR4_INCLUDE_DIR} 34 ${CMAKE_SOURCE_DIR}/src/generated 35)

Agrega dos directorios al search path de includes solo para este target (de ahí el PRIVATE):

  • Línea 33: el directorio del runtime ANTLR, para que #include "antlr4-runtime.h" funcione.
  • Línea 34: el directorio donde están PatitoLexer.h y PatitoParser.h.
37target_link_libraries(patito PRIVATE ${ANTLR4_LIB})

Enlaza el binario contra la librería del runtime ANTLR. Sin esto, los símbolos de ANTLRInputStream, BaseErrorListener, etc., quedarían sin resolver al linkear y el build fallaría con undefined symbols.

tests/run_tests.sh

tests/run_tests.sh
Runner del test plan. Ejecuta cada programa de prueba y compara el exit code con el esperado, generando un reporte PASS/FAIL.
1#!/usr/bin/env bash

Shebang. Le dice al sistema qué intérprete usar para correr el script. /usr/bin/env bash en lugar de /bin/bash directo es más portable, ya que en algunos sistemas Bash no está en /bin.

2# Runner del Test-Plan de Patito (Etapa 1). 3# Ejecuta cada programa con ./build/patito y verifica el exit code esperado: 4# tests/valid/* -> debe parsear OK (exit 0) 5# tests/invalid/* -> debe fallar (exit 1)

Comentario de cabecera con el contrato del runner: qué hace y qué espera. En Bash los comentarios empiezan con #.

7set -u

Activa el modo "strict variables": si el script intenta usar una variable que no existe, aborta inmediatamente. Sin esto, un typo en un nombre de variable sería ignorado silenciosamente y daría resultados incorrectos.

8ROOT="$(cd "$(dirname "$0")/.." && pwd)" 9PATITO="$ROOT/build/patito"

Línea 8: calcula la ruta absoluta de la raíz del proyecto. Desglose: $0 es el path del script en sí, dirname "$0" es el directorio donde vive (tests/), cd .../.. sube un nivel a la raíz, pwd imprime la ruta absoluta. Así el script funciona aunque lo llames desde cualquier directorio.

Línea 9: arma la ruta del binario que vamos a ejecutar.

11if [ ! -x "$PATITO" ]; then 12 echo "Binario no encontrado en $PATITO. Compila con: cmake --build build" 13 exit 2 14fi

Verifica que el binario exista y sea ejecutable (-x). Si no, imprime cómo construirlo y aborta con código 2 (error de uso). Esto evita el caso confuso donde corres el runner sin haber compilado y sale "FAIL" en cada test sin razón clara.

16PASS=0 17FAIL=0 18FAILED_FILES=()

Inicializa los contadores y un arreglo vacío para guardar la lista de archivos fallidos. FAILED_FILES=() declara un arreglo Bash vacío.

20run_case() { 21 local file="$1" 22 local expected="$2" # 0 o 1 23 "$PATITO" "$file" >/dev/null 2>&1 24 local got=$?

Define la función que ejecuta un caso de prueba.

  • Líneas 21–22: local declara variables locales a la función. $1 es el primer argumento (path al archivo) y $2 el segundo (exit code esperado).
  • Línea 23: ejecuta el binario sobre el archivo. >/dev/null descarta stdout y 2>&1 redirige stderr al mismo lugar (también descartado). No quiero ver el output del programa, solo el exit code.
  • Línea 24: $? es el exit code del último comando ejecutado.
25 # Normalizar: cualquier exit !=0 cuenta como fallo de parseo (1) 26 if [ $got -ne 0 ]; then got=1; fi 27 if [ $got -eq $expected ]; then 28 printf " PASS %-50s (esperado=%d, obtenido=%d)\n" "$(basename "$file")" "$expected" "$got" 29 PASS=$((PASS+1)) 30 else 31 printf " FAIL %-50s (esperado=%d, obtenido=%d)\n" "$(basename "$file")" "$expected" "$got" 32 FAIL=$((FAIL+1)) 33 FAILED_FILES+=("$file") 34 fi 35}

Línea 26: el binario puede salir con 0 (OK), 1 (errores de parseo) o 2 (error de uso). Para fines de comparación, todo lo que no sea 0 lo trato como 1, simplificando la lógica.

Líneas 27–34: si el exit observado coincide con el esperado, es PASS, si no, es FAIL. printf con %-50s alinea el nombre del archivo en una columna fija de 50 caracteres para que el reporte se vea como tabla.

Línea 33: el caso fallido se agrega al arreglo de fallos para reportarlo al final.

37echo "========================================" 38echo " VALIDOS (deben parsear OK, exit=0)" 39echo "========================================" 40for f in "$ROOT"/tests/valid/*.patito; do 41 [ -e "$f" ] || continue 42 run_case "$f" 0 43done

Loop de los casos válidos. "$ROOT"/tests/valid/*.patito es un glob que se expande a la lista de archivos. Línea 41 ([ -e "$f" ] || continue) protege el caso degenerado donde el glob no encuentra nada (entonces $f contiene literalmente *.patito), saltándose esa "iteración fantasma".

Cada archivo se pasa a run_case con expected=0 (debe parsear OK).

45echo "" 46echo "========================================" 47echo " INVALIDOS (deben fallar, exit=1)" 48echo "========================================" 49for f in "$ROOT"/tests/invalid/*.patito; do 50 [ -e "$f" ] || continue 51 run_case "$f" 1 52done

Lo mismo pero con expected=1: estos programas tienen un error a propósito y deben provocar que el binario salga con 1.

54TOTAL=$((PASS+FAIL)) 55echo "" 56echo "========================================" 57echo " RESUMEN: $PASS / $TOTAL pasaron" 58echo "========================================" 59if [ $FAIL -gt 0 ]; then 60 echo "Fallidos:" 61 for f in "${FAILED_FILES[@]}"; do 62 echo " - $f" 63 done 64 exit 1 65fi 66exit 0

Resumen final:

  • Línea 54: cuenta total de tests ejecutados.
  • Líneas 55–58: imprime el resumen "X / Y pasaron".
  • Líneas 59–65: si hubo fallos, lista cada archivo fallido y sale con código 1 para que un CI pueda detectar la falla.
  • Línea 66: si todo pasó, exit 0. Sirve para encadenar este runner con otros scripts (por ejemplo, en un futuro pre-commit hook).
