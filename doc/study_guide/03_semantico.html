3.2 Registros de datos

VarInfo

class VarInfo:
    """Info de una variable: tipo, linea, direccion virtual."""
    def __init__(self, tipo, linea, direccion=None):
        self.tipo = tipo           # 'entero', 'flotante'
        self.linea = linea         # donde se declaro
        self.direccion = direccion # direccion virtual asignada

ParamInfo

class ParamInfo:
    """Info de un parametro: nombre, tipo, direccion. El orden importa."""
    def __init__(self, nombre, tipo, direccion=None):
        self.nombre = nombre
        self.tipo = tipo
        self.direccion = direccion

FuncInfo

class FuncInfo:
    def __init__(self, tipo_retorno, linea):
        self.tipo_retorno = tipo_retorno  # 'entero', 'flotante', 'nula' o 'programa'
        self.params = []                  # lista de ParamInfo en orden
        self.variables = {}               # {nombre -> VarInfo} de locales + params
        self.linea = linea
        self.cuad_inicio = None           # Etapa 4 lo llena
        self.recursos = None              # Etapa 5 lo llena (snapshot)
        self.dir_retorno = None           # extension retorna (solo entero/flotante)

El programa principal también se registra como un FuncInfo (con tipo_retorno="programa"). Esto reusa la misma estructura y evita casos especiales en el código.

3.3 AsignadorMemoria

Asigna direcciones virtuales por (segmento × tipo). 11 segmentos (después de la extensión retorna):

_BASE = {
    ("global",  "entero"):     1000,
    ("global",  "flotante"):   2000,
    ("global",  "bool"):       3000,
    ("local",   "entero"):     5000,
    ("local",   "flotante"):   6000,
    ("local",   "bool"):       7000,
    ("temp",    "entero"):     9000,
    ("temp",    "flotante"):  10000,
    ("temp",    "bool"):      11000,
    ("retorno", "entero"):    17000,   # NUEVO (extension retorna)
    ("retorno", "flotante"):  18000,   # NUEVO
}

Huecos intencionales en 4000, 8000, 12000, 16000 (buffer para futuras extensiones). 13000-15999 lo usa TablaConstantes.

Métodos

MétodoQué hace
reset_scope_local()Resetea contadores de local y temp. Llamado en cada enterFuncs + al entrar a main.
nueva_global(tipo)Asigna y avanza contador global.
nueva_local(tipo)Asigna y avanza contador local.
nuevo_temporal(tipo)Asigna y avanza contador temporal.
nueva_retorno(tipo)Asigna y avanza contador de retorno. No se resetea.
snapshot_recursos()Devuelve {(seg, tipo) -> count} de cuanto se usó en el scope. Para la VM.

Los segmentos global y retorno nunca se resetean. Los segmentos local y temp SÍ (en cada enterFuncs + al entrar a main). Eso permite que cada función parta sus locales/temps en su _BASE.

3.4 Helpers para aplanar gramática

La gramática de Patito usa recursión derecha para las listas (porque ANTLR4 no necesita LL recursión para esto). Eso da estructuras como:

idLista : ID idResto ;
idResto : COMA ID idResto | epsilon ;

Si tienes a, b, c, d, el árbol queda anidado: idLista(a, idResto(b, idResto(c, idResto(d, eps)))). Para procesar todos los IDs de un jalón, dos helpers aplanan estas estructuras:

def _ids_de_idLista(idLista_ctx):
    """Aplana idLista + idResto en lista de TerminalNode ID."""
    if idLista_ctx is None:
        return []
    ids = [idLista_ctx.ID()]
    resto = idLista_ctx.idResto()
    while resto is not None and resto.ID() is not None:
        ids.append(resto.ID())
        resto = resto.idResto()
    return ids

3.5 SemanticAnalyzer — el estado

class SemanticAnalyzer(PatitoListener):
    def __init__(self):
        self.func_dir = {}              # {nombre -> FuncInfo}
        self.scope_actual = None        # puntero al FuncInfo actual
        self.nombre_programa = None
        self.errores = 0
        self.en_func_duplicada = False  # flag para abortar PNs en error
        self.mem = AsignadorMemoria()
        self.cte = TablaConstantes()
        self.gen = GeneradorCuadruplos(self.mem, self.cte)
        self.pila_llamadas = []         # contexto de llamadas (anidables)

scope_actual es un puntero al FuncInfo activo. Cambia en:

  • enterPrograma: scope_actual = func_dir[programa]
  • enterFuncs: scope_actual = nueva_func_info
  • exitFuncs: scope_actual = func_dir[programa] (regresa al global)
Todas las decisiones que dependen del scope (registrar var como global o local, buscar var en variables locales) usan este puntero.

Helper _buscar_var

def _buscar_var(self, nombre):
    """Busca variable: primero local, luego global. Devuelve VarInfo o None."""
    if nombre in self.scope_actual.variables:
        return self.scope_actual.variables[nombre]
    programa = self.func_dir[self.nombre_programa]
    if self.scope_actual is not programa and nombre in programa.variables:
        return programa.variables[nombre]
    return None

Implementa el lookup de scope clásico: local primero, luego global. Si la var no está ni en local ni en global, devuelve None — el caller reporta error "variable no declarada".

3.7 Tips de examen

"¿Por qué _buscar_var busca primero local y luego global?"

Para implementar shadowing: si una función declara una variable x y existe una global con el mismo nombre, la local "tapa" a la global dentro del scope de la función. Estilo léxico/estático clásico.

"¿Cuándo se llama reset_scope_local()?"

En 2 lugares:

  1. enterFuncs: al entrar a una nueva función, sus contadores locales/temp parten en sus _BASE.
  2. enterCuerpo cuando el padre es ProgramaContext (es main): para que main no herede contadores de la última función.
NO en exitFuncs — ahí se hace snapshot ANTES de cualquier reset (para preservar los recursos).

"¿Por qué exitExpresion hace dispatch a helpers?"

Para evitar duplicar código. Una expresión puede aparecer en 4 contextos: condición de si/mientras, argumento de llamada, RHS de asignación, o argumento de escribe. En vez de tener 4 PNs distintos, exitExpresion revisa el padre y delega: condición -> _manejar_condicion, argumento -> _manejar_argumento. Asigna y escribe son PNs propios.

Diferencia clave entre scope_actual y self.func_dir[programa]: el primero es dinámico (apunta al FuncInfo actualmente activo durante la compilación), el segundo es estático (siempre apunta al FuncInfo del programa principal). _buscar_var usa ambos: primero el actual (locales), luego el global.
